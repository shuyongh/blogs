目录结构

    ─ ${application}-${version}
      ├── bin
      ├── doc
      │   ├── html
      │   ├── man[1-9]
      │   ├── pdf
      │   ├── internal
      │   └── examples
      ├── ebin
      │   └── ${application}.app
      ├── include
      ├── priv
      │   ├── lib
      │   └── bin
      └── src
	  |__ Emakefile
  • src - 可选。包含应用程序自身使用的 Erlang 源代码和内部 include 文件。
  • ebin - 必需。包含 Erlang 对象代码，即 .beam 文件。 .app 文件也必须放置在此处。
  • priv - 可选。用于应用程序特定的文件。 code:priv_dir/1 用于访问此目录。
  • priv/lib - 推荐。如果应用程序使用了任何共享对象文件，例如 NIF 或链接的驱动程序，应将这些文件放置在此处。
  • priv/bin - 推荐。任何由应用程序使用的可执行文件，例如端口程序，应放置在此处。
  • include - 可选。用于公共包含文件，这些文件必须从其他应用程序中访问。
  • bin - 可选。任何应用程序的可执行文件，如 escripts 或 shell 脚本，应放置在此处。
  • doc - 可选。任何已发布的文档应放在这里的子目录中。

应用启动与停止

启动单个应用程序

  • 编译
erl -make
  • 指定搜索路径启动shell
erl -pa /ebin
  • 加载应用程序
application:load(demo)
  • 启动应用程序
application:start(demo)
  • 重新编译应用程序
make:all([load]).
  • 停止应用程序
application:stop(demo)

启动多个应用程序

  • 确保两个应用程序在同级目录
  • 启动erlang
    • ERL_LIBS变量是您环境中定义的一个特殊变量，它允许您指定 Erlang 在哪里可以找到 OTP 应用程序。然后，VM 能够自动在其中查找ebin/目录
erl -env ERL_LIBS "."
  • 加载应用程序
application:load(demo1)
application:load(demo2)
  • 启动应用程序
application:start(demo1)
application:start(demo2)

事件处理

事件处理器

-module(event_handler) .
-export([make/l, add_handler/2, event/2]).

%%-------------------------------------------
%% 创建一个进程，用于制作一个名为Name的新事件处理器
%% 处理函数是no_op，代表不对事件做任何处理
%%-------------------------------------------
make (Name) ->
	register(Name, spawn(fun() -> my_handler(fun no_op/1) end) ) .

%%-------------------------------------------
%% 给名为Name的事件处理器添加一个处理函数Fun。
%% 当触发事件时，事件处理器就会执行Fun(X)。
%%-------------------------------------------
add_handler(Name, Fun) -> Name ! {add, Fun}.

%%-------------------------------------------
%% 触发事件处理器中的处理函数
%%-------------------------------------------
event(Name, X) -> Name ！ {event, X}.

my_handler(Fun) ->
    receive
        {add, Fun1} ->
        	my_handler(Fun1) ;
        {event，Any} ->
        	(catch Fun(Any)),
        	my_handler(Fun)
    end.
no_op(_) -> void.

回调模块

要让事件处理器能做点什么，必须编写一个回调模块并把它安装到事件处理器里。

-module(motor_controller) .
-export([add_event_handler/0]).
add_event_handler() ->
	event_handler:add_handler(errors， fun controller/1).
controller(too_hot) ->
	io:format(*Turn off the motor~n");
controller(x)->
	io:format("~w ignored event: ~p~n",[?MODULE, X]).

错误记录

OTP系统自带一个可定制的错误记录器。可以从三个角度看待错误记录器：

  • 程序员视角关心程序员为了记录错误而在代码里中做的函数调用；

  • 配置视角关心错误记录器在何处以及如何保存数据；

  • 报告视角关心对已发生错误的分析。

error_logger模块

Erlang/OTP 提供的一个日志记录工具，用于记录系统运行中的错误、警告、信息等日志。

方法名描述示例使用
error_logger:log/1记录日志消息（不指定日志级别）error_logger:log({error, "Error occurred"})
error_logger:info_msg/1记录信息级别的日志消息error_logger:info_msg("Informational message")
error_logger:warning_msg/1记录警告级别的日志消息error_logger:warning_msg("Warning message")
error_logger:error_msg/1记录错误级别的日志消息error_logger:error_msg("Error message")
error_logger:debug_msg/1记录调试级别的日志消息error_logger:debug_msg("Debug message")
error_logger:format/2记录日志消息并使用格式化字符串error_logger:format("User ~p logged in", [User])
error_logger:notify/1向错误日志系统发送通知error_logger:notify({error, "Fatal error occurred"})
error_logger:drop/0停止错误日志记录功能error_logger:drop()
error_logger:log_report/1记录复杂的日志报告（如堆栈信息等）error_logger:log_report({stack_trace, Stack})

配置错误记录器

有多种方式可用来配置错误记录器。

  • 可以让Erlang shell显示所有错误（如果没有特别设置的话就是默认值），
  • 也可以把shell里报告的所有错误写入一个格式化文本文件。
  • 还可以创建一个滚动日志（rotating log）。
    • 可以把滚动日志看作是一个大型循环缓冲区，内含错误记录器生成的消息。新消息进来后会被附加到日志的末尾，如果日志满了，最早的条目就会被删除。
    • 滚动日志极其有用。你决定日志应当占据多少个文件，以及每个日志文件能有多大，然后系统负责在一个大型循环缓冲区里删除旧日志文件和创建新文件。可以调整日志的大小来保存最近几天的操作记录，这通常足以应付大多数用途了。

标准错误记录器

  • 创建一个适合进行程序开发的环境，只提供一种简单的错误记录形式。
    • 不带启动参数的erl命令就等于erl -boot start_clean。
erl -boot start_clean
  • 创建一个适合运行生产系统的环境。系统架构支持库（System Architecture Support Libraries，简称SASL）将负责错误记录和过载保护等工作。
    • 日志文件的配置最好通过配置文件实现，因为没人能记住记录器的全部参数。
erl -boot start_sasl -config 配置文件名（不要后缀后，该文件是以.config结尾）

无配置SASL

  • 错误是在Erlang shell里报告的。错误的报告位置由错误记录器配置决定。

  • 错误记录器会生成多种报告类型。这三种报告会自动生成，程序员无须做任何事。

    • 监控器报告：这些报告会在OTP监控器启动或停止被监控进程时生成。

    • 进度报告：这些报告会在OTP监控器启动或停止时生成。

    • 崩溃报告：如果某个被OTP行为启动的进程因为normal或shutdown以外的原因终止，这些报告就会生成。

  • 可以显式调用error_logger模块里的方法来生成三种类型的日志报告。这让我们能够记录错误、警告和信息消息。

sys.config

sys.config 是 Erlang 系统的主要配置文件之一，用于配置系统的核心参数和模块。这个文件通常用于配置应用的行为以及系统的全局设置。

系统和应用配置：sys.config 用来设置 Erlang 系统和各个 Erlang 应用的配置项。这些配置项可以包括日志记录、节点名称、应用的启动顺序、环境变量等。

全局配置：一些全局配置选项（如 SASL 日志、Erlang 内核的配置等）可以在 sys.config 中进行设置。

  • 该文件位置优先级
    1. 该文件默认位置erlang安装路径\releases\<version>\sys.config
    2. 项目目录下.erlang文件中配置的sys.config路径
    3. 项目目录下的sys.config
    4. 启动erlang时，使用-config指定文件位置
%% sys.config
[
  {kernel, [
      {logger, true},               % 启用日志记录
      {logger_level, info},         % 设置日志级别
      {cookie, "secretcookie"}      % 设置分布式 Erlang 的 cookie
  ]},
  {sasl, [
      {error_logger, file},         % 设置错误日志的输出目标
      {log_dir, "/var/log/erlang"}  % 设置日志文件的存储目录
  ]}
].

``sasl.config`

SASL（Simple Authentication and Security Layer）是 Erlang 系统中的一个子系统，负责处理错误日志、监控、系统诊断等。sasl.config 专门用于配置与 SASL 相关的参数，尤其是与错误日志相关的配置。

SASL 配置：sasl.config 主要用于配置与 SASL 相关的行为，特别是错误日志的记录、系统诊断、错误报告的目标（如日志文件、控制台或远程目标）。

错误日志：SASL 通过 error_logger 记录系统中的错误和异常信息。你可以在 sasl.config 中配置日志输出目标、日志级别等。

[
  {sasl, [
      {error_logger, file},             % 错误日志输出到文件
      {log_level, info},                % 设置最低日志级别为 info
      {log_dir, "/var/log/erlang"}      % 设置日志文件的存储目录
  ]}
].

%% elog.config

    ]}].
