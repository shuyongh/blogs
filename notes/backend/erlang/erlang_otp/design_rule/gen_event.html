gen_event行为

约 1423 字大约 5 分钟

2025-03-22

事件处理原则

在 OTP 中，事件管理器是一个可以发送事件的命名对象。事件可以是错误、警报或需要记录的信息。

在事件管理器中，可以安装零个、一个或多个事件处理器。当事件管理器被通知有关事件时，该事件由所有已安装的事件处理器进行处理。

事件管理器作为进程实现，每个事件处理器作为回调模块实现

事件管理器本质上维护一个 {Module, State} 对列表，其中每个 Module 是一个事件处理器， State 是该事件处理器的内部状态。

服务端(回调模块)---事件处理器

gen_event module                   Callback module
----------------                   ---------------
gen_event:start
gen_event:start_monitor
gen_event:start_link       ----->  -

gen_event:add_handler
gen_event:add_sup_handler  ----->  Module:init/1

gen_event:notify
gen_event:sync_notify      ----->  Module:handle_event/2

gen_event:send_request
gen_event:call             ----->  Module:handle_call/2

-                          ----->  Module:handle_info/2

gen_event:delete_handler   ----->  Module:terminate/2

gen_event:swap_handler
gen_event:swap_sup_handler ----->  Module1:terminate/2
                                   Module2:init/1

gen_event:which_handlers   ----->  -

gen_event:stop             ----->  Module:terminate/2

-                          ----->  Module:code_change/3

init/1

参数

  • Args: 客户端调用start_link时发送的参数

返回值

  • {ok,State} State 将直接传递到进程的主循环中，作为稍后保留的状态
  • {ok,State,TimeOut} 如果在截止时间之前没有收到消息，则会向服务器发送一个特殊的（原子 timeout）消息，应通过 handle_info/2处
  • {ok,State,hibernate} 如果你确实希望进程在获得回复之前花费很长时间，并且担心内存，你可以将 hibernate 原子添加到元组中。
  • {stop,Reason} 在初始化过程中出现错误时

handle_event/2

处理异步消息

handle_event(Event, State) -> result()

参数

  • Event
  • State

返回值

  • {ok,NewState} 只是更新自己的状态，不会回复任何人
  • {ok,NewState,hibernate} 整个事件管理器将被置于休眠状态
  • remove_handler 将处理程序从管理器中删除
  • {swap_handler,args1,NewState,NewHandler,Args2} 删除当前的事件处理程序，并用一个新的事件处理程序替换它
    • 这是通过首先调用CurrentHandler:terminate(Args1, NewState) 并删除当前的处理程序，然后通过调用NewHandler:init(Args2, ResultFromTerminate) 添加一个新的处理程序来完成的

handle_call/2

处理同步消息

handle_call(Request,  State) -> result().

参数

  • Request
  • State

返回值

  • {ok, Reply, NewState}
  • {ok, Reply, NewState, hibernate}
  • {remove_handler, Reply}
  • {swap_handler, Reply, Args1, NewState, Handler2, Args2}

handle_info/2

只处理带外消息，例如退出信号、使用! 运算符直接发送到事件管理器的消息等。

handle_info(_, State) ->
    {ok, State}.

code_change/3

它针对每个单独的事件处理程序

code_change(_OldVsn, State, _Extra) ->
    {ok, State}.

terminate/2

init的相反操作

terminate(_Reason, _State) -> ok.

模板

-module(template).
-behaviour(gen_event).

-export([init/1, handle_event/2, handle_call/2, handle_info/2, code_change/3,
   terminate/2]).

init([]) ->
    {ok, []}.

handle_event(_, State) ->
    {ok, State}.

handle_call(_, State) ->
    {ok, ok, State}.

handle_info(_, State) ->
    {ok, State}.

code_change(_OldVsn, State, _Extra) ->
    {ok, State}.

terminate(_Reason, _State) ->
    ok.

客户端

开始一个事件管理器

启动一个事件管理器

如果事件管理器是监督树的一部分，意味着它是由监督器启动的，则必须使用gen_event:start_link/1。还有一个函数gen_event:start/1，用于启动不属于监督树的自立式事件管理器。

-type emgr_name() :: {'local', atom()} | {'global', term()}
                   | {'via', atom(), term()}.
-type options() :: [{'timeout', timeout()}
                   | {'debug', [debug_flag()]}
                   | {'spawn_opt', [proc_lib:start_spawn_option()]}
                   | {'hibernate_after', timeout()}].
-type start_ret() :: {'ok', pid()} | {'error', term()}.

-spec gen_event:start_link(EventMgrName :: emgr_name()) -> start_ret();
                (Options :: options()) -> start_ret().

添加事件处理器

向指定事件管理器中添加事件处理器

gen_event:add_handler(Pid, Handler, Args).

参数

  • Pid 事件管理器的标识
  • Handler 向事件管理器中添加的事件处理器模块名字
  • Args 发送给事件管理器的参数

返回值

  • 事件管理器调用回调函数Handler:init([])，其中参数[]是add_handler的第三个参数。init/1预期返回{ok, State}，其中State是事件处理器的内部状态。

通知事件

向事件管理器发送事件。是异步的，

gen_event:sync_notify/2，它是同步的

gen_event:notify(EventMgrRef, Event)->ok.

参数

  • EventMgrRef
  • Event

返回值

  • 事件被转换为消息并发送给事件管理器。当接收到事件时，事件管理器按照它们添加的顺序，为每个已安装的事件处理器调用handle_event(Event, State)。该函数预期返回一个元组{ok,State1}，其中State1是事件处理器状态的新值。

提示

handle_event/2 仍然是异步的。这里的想法是，函数调用只在所有事件处理程序都看到并处理了新消息后才会返回。在此之前，事件管理器将通过不回复来一直阻塞调用进程。

删除事件处理器

向指定事件管理器中删除事件处理器

gen_event:delete_handler(Pid, Handler, []).

参数

  • Pid 事件管理器的标识
  • Handler 向事件管理器中删除的事件处理器名字
  • Args 发送给事件管理器的参数

返回值

  • 事件管理器调用回调函数 Handler:terminate([], State) ，其中参数 [] 是 delete_handler 的第三个参数。terminate/2 应与 init/1 相反，并执行任何必要的清理。其返回值被忽略。

此函数向注册为 error_man 的事件管理器发送消息，告知其删除事件处理程序 terminal_logger。

停止

当事件管理器停止时，它会给每个已安装的事件处理器机会通过调用terminate/2来清理，就像删除处理器时一样。

在一个监督树

如果事件管理器是监督树的一部分，则不需要停止函数。事件管理器将由其监督者自动终止。具体如何执行由监督者中设置的关闭策略定义。

独立事件管理器

事件管理器也可以通过调用以下方式停止：

1> gen_event:stop(error_man).
