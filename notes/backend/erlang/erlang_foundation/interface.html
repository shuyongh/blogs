接口技术

约 6463 字大约 22 分钟

2025-03-22

在 Erlang 中，"接口技术" 通常是指如何使 Erlang 程序与外部世界进行交互，包括与其他编程语言、外部服务、硬件设备等进行通信

Erlang 运行时系统内置了两种互操作机制，即 分布式 Erlang、端口 和 NIF。端口的一种变体是链接式驱动。

问题引入

一个常见的互操作性场景是，当你想在你的 Erlang 程序中整合一段解决复杂问题的代码时。例如，假设你有以下 C 函数，你想从 Erlang 中调用它们

/* complex.c */

int foo(int x) {
  return x+1;
}

int bar(int y) {
  return y*2;
}

为了便于阅读，这些函数被刻意保持得尽可能简单。

从 Erlang 的角度来看，最好能够调用 foo 和 bar，而无需关心它们是 C 函数

% Erlang code
...
Res = complex:foo(X),
...

在这里，与 C 的通信隐藏在 complex.erl 的实现中。在接下来的章节中，将展示如何使用不同的互操作性机制来实现此模块。

分布式 Erlang

注

通过给 Erlang 运行时系统命名，将其变成分布式 Erlang 节点。分布式 Erlang 节点可以连接并监控其他节点。它还可以在其他节点上创建进程。不同节点间进程的消息传递和错误处理是透明的。分布式 Erlang 系统中提供了一些有用的 STDLIB 模块。例如，global，它提供全局命名注册。分布机制使用 TCP/IP 套接字实现。

何时使用： 分布式 Erlang 主要用于 Erlang-Erlang 通信。它也可以用于 Erlang 和 C 之间的通信。 C 程序作为 C 节点实现

C节点

注

一个使用 Erl 接口函数来设置连接并与分布式 Erlang 节点通信的 C 程序被称为 C 节点 ，或称为 隐藏节点 。C 节点的主要优势在于，从 Erlang 程序员的视角来看，通信极为简单，因为 C 程序表现得像一个分布式 Erlang 节点。

何时使用： C 节点通常可用于设备处理器（与控制处理器相对），在内存限制或应用程序特性方面，C 比 Erlang 更优，或者两者兼而有之。

端口和链接驱动程序

提示

从 Erlang 的角度来看，端口提供了与外部世界通信的基本机制。端口为外部程序提供了一个面向字节的接口。创建端口后，Erlang 可以通过发送和接收 字节列表 或 二进制数据（而不是 Erlang 项）与它通信。

链接式驱动：外部程序驻留在与 Erlang 运行时系统不同的操作系统进程中。在某些情况下，这是不可接受的。例如，考虑具有非常严格时间要求的驱动程序。因此，可以按照一定的原则用 C 编写程序，并将其动态链接到 Erlang 运行时系统。这被称为 链接式驱动。

端口和进程的关系：Erlang 中的端口本质上是一个特殊的进程，但它不由 Erlang 虚拟机（BEAM）调度，它通过消息与 Erlang 进程通信。端口进程通常用于与外部程序进行双向通信。

**何时使用 ： 端口可用于 Erlang 程序和其他程序在同一台计算机上运行的各种互操作情况。

Erlang 外部项格式是将 Erlang 项表示为字节数据序列，即二进制。两种表示之间的转换使用以下 BIFs 完成

%% Erlang 项 -> 二进制
term_to_binary(Term) -> Binary
%% 二进制 -> Erlang 项
binary_to_term(Binary) -> Term

常用方法

方法说明示例
open_port(Args, Options)打开一个端口，启动外部程序，并返回一个端口标识符
port_command(Port, Data)向端口发送数据
port_info(Port)获取有关端口的信息, 接收端口数据
flush()接收从端口发送到当前进程的所有消息
port_close(Port)关闭端口

在 Erlang 中，端口（Port）是与外部程序交互的机制，使用端口时会涉及多个常用的函数和参数。以下是对 Erlang 中端口相关的常用函数及其参数的详细介绍。

1. open_port/2

open_port/2 用于打开一个端口，启动外部程序，并返回一个端口标识符 Port。这个端口标识符用于与外部程序进行交互。

函数签名

-spec open_port(PortName, PortSettings) -> port()
                   when
                       PortName ::
                           {spawn, Command :: string() | binary()} |
                           {spawn_driver, Command :: string() | binary()} |
                           {spawn_executable, FileName :: file:name_all()} |
                           {fd, In :: non_neg_integer(), Out :: non_neg_integer()},
                       PortSettings :: [Opt],
                       Opt ::
                           {packet, N :: 1 | 2 | 4} |
                           stream |
                           {line, L :: non_neg_integer()} |
                           {cd, Dir :: string() | binary()} |
                           {env,
                            Env :: [{Name :: os:env_var_name(), Val :: os:env_var_value() | [] | false}]} |
                           {args, [string() | binary()]} |
                           {arg0, string() | binary()} |
                           exit_status | use_stdio | nouse_stdio | stderr_to_stdout | in | out |
                           binary | eof |
                           {parallelism, Boolean :: boolean()} |
                           hide |
                           {busy_limits_port, {non_neg_integer(), non_neg_integer()} | disabled} |
                           {busy_limits_msgq, {non_neg_integer(), non_neg_integer()} | disabled}.

参数

PortName 用来指定外部进程或设备的启动方式。它可以是以下几种格式：

  • {spawn, Command :: string() | binary()}

    • 描述：通过 spawn 启动外部命令或脚本。

    • 参数：Command 是要执行的命令，可以是字符串或二进制格式。

  • {spawn_driver, Command :: string() | binary()}

    • 描述：启动外部命令，作为一个 Erlang 驱动程序来处理 I/O。

    • 参数：Command 是要启动的驱动命令。

  • {spawn_executable, FileName :: file:name_all()}

    • 描述：启动外部可执行文件。

    • 参数：FileName 是文件路径，指定要执行的可执行文件。

  • {fd, In :: non_neg_integer(), Out :: non_neg_integer()}

    • 描述：通过文件描述符（FD）创建端口。这用于直接与文件或管道进行通信。

    • 参数：In 和 Out 是文件描述符，分别表示输入和输出通道。

PortSettings 是一组选项，用于配置如何与端口进行交互。它是一个包含多个配置项的列表。

  • {packet, N :: 1 | 2 | 4}

    • 描述：设置如何通过端口发送数据包。

    • 参数：N 是数据包的大小，允许值为 1、2 或 4。

  • stream

    • 描述：设置端口为流模式，这意味着数据将按字节流的方式发送和接收。

  • {line, L :: non_neg_integer()}

    • 描述：设置端口每次接收的最大字节数 L。通常用于处理文本流数据时，按行读取。

  • {cd, Dir :: string() | binary()}

    • 描述：指定端口启动时的工作目录（current directory）。

    • 参数：Dir 是一个字符串或二进制表示的目录路径。

  • {env, Env :: [{Name :: os:env_var_name(), Val :: os:env_var_value() | [] | false}]}

    • 描述：为外部进程设置环境变量。

    • 参数：Env 是一个列表，包含环境变量的名称和值。

  • {args, [string() | binary()]}

    • 描述：指定要传递给外部命令的参数。

    • 参数：args 是一个字符串或二进制列表，表示要传递的命令行参数。

  • {arg0, string() | binary()}

    • 描述：指定外部命令的第一个参数（通常是程序名称）。

    • 参数：arg0 是一个字符串或二进制，表示第一个参数。

  • exit_status

    • 描述：表示是否捕获外部进程的退出状态。

  • use_stdio

    • 描述：表示标准输入/输出应该用于与端口的通信。

  • nouse_stdio

    • 描述：表示不要使用标准输入/输出与端口通信。

  • stderr_to_stdout

    • 描述：将标准错误（stderr）重定向到标准输出（stdout）。

  • in | out

    • 描述：分别指定端口的输入或输出通道。

  • binary

    • 描述：表示数据将以二进制格式进行传输。

  • eof

    • 描述：表示数据流的结尾，通常用于处理文本文件或流。

  • {parallelism, Boolean :: boolean()}

    • 描述：表示是否启用并行化处理。

    • 参数：Boolean 可以是 true 或 false。

  • hide

    • 描述：表示端口应隐藏其输出。

  • {busy_limits_port, {non_neg_integer(), non_neg_integer()} | disabled}

    • 描述：表示设置端口的忙碌限制（如队列长度等）。disabled 表示没有限制。

  • {busy_limits_msgq, {non_neg_integer(), non_neg_integer()} | disabled}

    • 描述：表示设置消息队列的忙碌限制。disabled 表示没有限制。

返回值

open_port/2 函数返回一个 port() 类型，表示与外部进程或设备的通信端口。你可以通过这个端口与外部进程进行消息交换。

3. port_info/1

port_info/1 用于获取关于端口的详细信息。它返回一个包含端口状态的元组或列表，这有助于检查端口的运行状态或调试。

函数签名：

port_info(Port) -> Info

参数：

  • Port: 端口标识符，通常是由 open_port/2 返回的值。

返回值：

port_info/1 返回的值是一个包含有关端口的详细信息的元组。常见的信息有：

  • {name, Program}: 端口对应的外部程序。
  • {status, Status}: 端口的当前状态（例如，{open, Program} 表示端口正在打开，或者 {closed} 表示端口已关闭）。
  • {exit_status, Code}: 外部程序退出时的状态码（如果适用）。
  • {data, Data}: 端口读取到的数据。

flush/0

flush/0 是用于接收从端口发送到当前进程的所有消息的函数。这在处理来自多个端口的消息时非常有用。

函数签名：

flush() -> Msgs

参数：

没有参数。

返回值：

flush/0 会返回当前进程消息队列中的所有消息。

Erlang 程序

注

Erlang 和 C 之间的所有通信都必须通过创建端口来建立。创建端口的 Erlang 进程被称为端口的 连接进程。所有进出端口的通信都必须经过连接进程。如果连接进程终止，端口也会终止（如果外部程序编写正确，外部程序也会终止）。

-module(complex1).
-export([start/1, stop/0, init/1]).
-export([foo/1, bar/1]).

start(ExtPrg) ->
    spawn(?MODULE, init, [ExtPrg]).
stop() ->
    complex ! stop.


% 实现 complex1:foo/1 和 complex1:bar/1。
% 两者都向 complex 进程发送消息并接收以下回复
foo(X) ->
    call_port({foo, X}).
bar(Y) ->
    call_port({bar, Y}).

call_port(Msg) ->
    complex ! {call, self(), Msg},
    receive
        {complex, Result} ->
            Result
    end.

% complex 进程执行以下操作
    % 1.将消息编码为字节序列。
    % 2.将其发送到端口。
    % 3.等待回复。
    % 4.解码回复。
    % 5.将其发送回调用者
loop(Port) ->
    receive
        {call, Caller, Msg} ->
            Port ! {self(), {command, term_to_binary(Msg)}},
            receive
                {Port, {data, Data}} ->
                    Caller ! {complex, binary_to_term(Data)}
            end,
            loop(Port);
        stop ->
            Port ! {self(), close},
            receive
                {Port, closed} ->
                    exit(normal)
            end;
        {'EXIT', Port, Reason} ->
            exit(port_terminated)
    end.


% 创建端口
init(ExtPrg) ->
    register(complex, self()),
    %设置为捕获退出，这使得能够检测外部程序的失败
    process_flag(trap_exit, true),
    %字符串 ExtPrg 是外部程序的名称，包括任何命令行参数
    %{packet,2}。此选项表示使用 2 字节的长度指示符来简化 C 和 Erlang 之间的通信。
    %Erlang 端口会自动添加长度指示符，但必须在外部 C 程序中显式完成此操作。
    Port = open_port({spawn, ExtPrg}, [{packet, 2}, binary]),
    loop(Port).

C 程序

注

在 C 端，有必要编写函数来从/向 Erlang 接收和发送带有 2 字节长度指示符的数据。默认情况下，C 程序应从标准输入（文件描述符 0）读取并写入标准输出（文件描述符 1） :::

/* erl_comm.c */

#include <stdio.h>
#include <unistd.h>

typedef unsigned char byte;

int read_exact(byte *buf, int len)
{
  int i, got=0;

  do {
      if ((i = read(0, buf+got, len-got)) <= 0){
          return(i);
      }
    got += i;
  } while (got<len);

  return(len);
}

int write_exact(byte *buf, int len)
{
  int i, wrote = 0;

  do {
    if ((i = write(1, buf+wrote, len-wrote)) <= 0)
      return (i);
    wrote += i;
  } while (wrote<len);

  return (len);
}

int read_cmd(byte *buf)
{
  int len;

  if (read_exact(buf, 2) != 2)
    return(-1);
  len = (buf[0] << 8) | buf[1];
  return read_exact(buf, len);
}

int write_cmd(byte *buf, int len)
{
  byte li;

  li = (len >> 8) & 0xff;
  write_exact(&li, 1);

  li = len & 0xff;
  write_exact(&li, 1);

  return write_exact(buf, len);
}

注意

请注意，stdin 和 stdout 用于缓冲输入/输出，不能用于与 Erlang 的通信。

  • 在 main 函数中，C 程序应监听来自 Erlang 的消息，并根据选定的编码/解码方案，使用第一个字节来确定要调用的函数，第二个字节作为函数的参数。然后将调用函数的结果发送回 Erlang。
/* port.c */

typedef unsigned char byte;

int main() {
  int fn, arg, res;
  byte buf[100];

  while (read_cmd(buf) > 0) {
    fn = buf[0];
    arg = buf[1];

    if (fn == 1) {
      res = foo(arg);
    } else if (fn == 2) {
      res = bar(arg);
    }

    buf[0] = res;
    write_cmd(buf, 1);
  }
}

注意

注意，C 程序在一个 while-循环中，检查 read_cmd/1 的返回值。这是因为 C 程序必须检测端口何时关闭和终止。

本地实现函数（Nifs）

注

NIFs 提供了一种使用链接式驱动程序将 C 代码链接到 Erlang 运行时系统的替代方案。NIFs 使得在与操作系统或其他外部库交互时， C 实现并提供正常 的Erlang 函数成为可能。

与使用端口驱动程序相比，NIF 是一种更简单、更高效地调用 C 代码的方式。

NIF 是一种用 C 语言实现而不是 Erlang 语言的函数。NIF 对调用者来说表现为任何其他函数。它们属于一个模块，并且像任何其他 Erlang 函数一样被调用。一个模块的 NIF 被编译并链接成一个动态可加载的共享库（UNIX 中的 SO，Windows 中的 DLL）。NIF 库必须在模块的 Erlang 代码运行时加载。

作为一个 NIF 库是动态链接到模拟器进程中的，这是从 Erlang 调用 C 代码最快的方式（与端口驱动程序并列）。调用 NIF 不需要进行上下文切换。但这也是最不安全的，因为 NIF 中的崩溃也会导致模拟器崩溃。

何时使用： 由于损坏的 NIF 可能导致与稳定性和安全相关的许多不同问题，建议如果可能的话使用外部端口。如果开销不可接受，那么 NIF 是与其他任何本地代码（无论是 C、C++还是 Rust）交互的好解决方案。

Erlang程序

注

即使一个模块的所有函数都是 NIF，仍然需要一个 Erlang 模块，原因有两个：

  • NIF 库必须由同一模块中的 Erlang 代码显式加载。
  • 一个模块中的所有 NIF 都必须有 Erlang 实现。

NIF 库通过调用 erlang:load_nif/2 加载，共享库的名称作为参数。第二个参数可以是任何将传递给库并用于初始化的项：

-module(demo).
-export([foo/1, bar/1]).
%% 指定哪些 模块内部定义的函数，可以作为 NIFs 加载的函数
%% 并非严格必要，但建议在任何加载 NIFs 的模块中使用 -nifs() 属性
-nifs([foo/1, bar/1]).
%% 指定一个将要运行的函数 当模块被加载时自动执行
-on_load(init/0).

init() ->
    ok = erlang:load_nif("./demo_nif", 0).
%%---------------------
%% 桩函数
%%---------------------
foo(_X) ->
    erlang:nif_error(nif_library_not_loaded).
bar(_Y) ->
    erlang:nif_error(nif_library_not_loaded).

加载 NIF 库会覆盖存根实现并导致调用 foo 和 bar 将被发送到 NIF 实现中。

提示

在桩函数中调用erlang:nif_error/1 此处调用内置函数erlang:nif_error/1有两个目的

  • 首先，如果Nif库未加载，则调用Nif函数会抛出运行时错误，在这种情况下，调用erlang:nif_error/1与erlang:error/1完全一样
  • 其次，是为了提示Dialyzer等代码分析工具这段 Erlang代码并不代表该函数的实际行为（要等到NIF库加载之后才能得以体现）

NIF库代码

注

模块中的 NIF 被编译并链接到一个共享库中。每个 NIF 都实现为一个普通的 C 函数。宏 ERL_NIF_INIT 与一个结构体数组共同定义了模块中所有 NIF 的名称、参数数量和函数指针。必须包含头文件 erl_nif.h。由于库是一个共享模块，而不是程序，因此不需要 main 函数。

传递给 NIF 的函数参数出现在一个名为 argv 的数组中，其中 argc 作为数组的长度，因此函数的元数。第 N 个参数 函数的参数可以通过 argv[N-1] 访问。NIFs 还接受一个环境参数，该参数作为一个不透明的句柄，需要传递给大多数 API 函数。环境包含有关调用 Erlang 进程的信息：

// 引入 Erlang NIF 库的头文件，提供所有 NIF 相关的函数和数据类型。
#include <erl_nif.h>

int foo(int x) {
    return x * 2;  // 示例实现
}

int bar(int y) {
    return y + 10;  // 示例实现
}
/**
 * foo_nif - 一个简单的 NIF 函数
 * static: 限制函数只在当前文件可见
 * ERL_NIF_TERM: Erlang 术语类型，表示 Erlang 数据
 * env: Erlang 环境指针，用于内存管理
 * argc: 参数个数
 */
static ERL_NIF_TERM foo_nif(ErlNifEnv* env, int argc, const ERL_NIF_TERM argv[])
{
    int x, ret;
    // 将 Erlang 术语转换为 C 的 int 类型
    //argv[0]: 第一个参数
    //&a: 将转换结果存储到变量 a 的地址
    if (!enif_get_int(env, argv[0], &x)) {
	      return enif_make_badarg(env);
    }
    ret = foo(x);
    return enif_make_int(env, ret);
}

static ERL_NIF_TERM bar_nif(ErlNifEnv* env, int argc, const ERL_NIF_TERM argv[])
{
    int y, ret;
    if (!enif_get_int(env, argv[0], &y)) {
	return enif_make_badarg(env);
    }
    ret = bar(y);
    return enif_make_int(env, ret);
}
/**
 * NIF 函数注册表
 * "foo": Erlang 中调用的函数名
 * 1: 函数参数个数（arity）
 * foo_nif: 对应的 C 函数指针
 */
static ErlNifFunc nif_funcs[] = {
    {"foo", 1, foo_nif},
    {"bar", 1, bar_nif}
};
/**
 * NIF 模块初始化
 * demo: Erlang 模块名
 * nif_funcs: 函数映射表
 * 后面的 NULL: 各种回调函数（加载、卸载、升级等），这里都不需要
 */
ERL_NIF_INIT(demo, nif_funcs, NULL, NULL, NULL, NULL)

在这里，ERL_NIF_INIT 有以下参数：

第一个参数必须是 Erlang 模块的 C 标识符名称。它将被宏转换成字符串。 第二个参数是包含每个 NIF 的名称、arity 和函数指针的 ErlNifFunc 结构数组的数组。 剩余的参数是指向回调函数的指针，这些回调函数可以用来初始化库。在本简单示例中，它们没有被使用，因此都设置为 NULL。 函数参数和返回值表示为类型 ERL_NIF_TERM 的值。在这里，使用像 enif_get_int 和 enif_make_int 这样的函数在 Erlang 项和 C 类型之间进行转换。如果函数参数 argv[0] 不是一个整数，enif_get_int 返回 false，在这种情况下，它通过抛出一个带有 enif_make_badarg 的 badarg-异常来返回。

C库Erl_Interface---ei库

注

Erl_Interface 库包含帮助您将用 C 和 Erlang 编写的程序集成的函数。Erl_Interface 中的函数支持以下功能：

  • 操作以 Erlang 数据类型表示的数据
  • 在 C 和 Erlang 格式之间转换数据
  • Erlang 数据类型的编码和解码，用于传输或存储
  • C 节点与 Erlang 进程之间的通信
  • 将 C 节点状态备份和恢复到 Mnesia

编译和链接

// myprog.c
/* 引入erl_interface函数 */
#include "ei.h"

编译：

$ cc -c -I$OTPROOT/lib/erl_interface-$EIVSN/include myprog.c

链接时：

  • 指定 libei.a 的路径为 -L$OTPROOT/lib/erl_interface-3.2.3/lib .
  • 指定库的名称为 -lei。
$ ld -L$OTPROOT/lib/erl_interface-$EIVSN/lib myprog.o -lei -o myprog

初始化库

在调用库中的其他任何函数之前，通过调用 ei_init() 函数恰好一次来初始化它。

编码、解码和发送 Erlang 项

注

在分布式 Erlang 节点之间传输的数据使用 Erlang 外部格式进行编码。因此，如果您想使用分布式协议在 C 程序和 Erlang 之间进行通信，您必须将 Erlang 项编码和解码为字节流。

详情查看ei模块

ei_x_buff buf;
ei_x_new(&buf);
int i = 0;
ei_x_encode_tuple_header(&buf, 2);
ei_x_encode_atom(&buf, "tobbe");
ei_x_encode_long(&buf, 3928);

构建术语

前一个示例可以通过使用 ei_x_format_wo_ver 函数来创建一个 Erlang 术语来简化：

ei_x_buff buf;
ei_x_new(&buf);
ei_x_format_wo_ver(&buf, "{~a,~i}", "tobbe", 3928);

以下示例更为复杂：

ei_x_buff buf;
int i = 0;
ei_x_new(&buf);
ei_x_format_wo_ver(&buf,
                   "[{name,~a},{age,~i},{data,[{adr,~s,~i}]}]",
                   "madonna",
                   21,
                  "E-street", 42);
ei_print_term(stdout, buf.buff, &i);
// 确保由 buf 指向的数据被释放。
ei_x_free(&buf);

连接到分布式 Erlang 节点

连接到分布式 Erlang 节点之前，您必须首先初始化 连接例程之一 ei_connect_init_* 函数，用于存储信息，例如主机名和节点名，以供后续使用：

int identification_number = 99;
int creation=1;
char *cookie="a secret cookie string"; /* An example */
const char* node_name = "einode@durin";
const char *cookie = NULL;
short creation = time(NULL) + 1;
ei_cnode ec;
ei_connect_init(&ec,
                node_name,
                cookie,
                creation);

初始化后，您设置与 Erlang 节点的连接。要指定要连接的 Erlang 节点，请使用 ei_connect_*() 函数族。以下示例设置连接，并生成有效的套接字文件描述符：

int sockfd;
const char* node_name = "einode@durin"; /* An example */
if ((sockfd = ei_connect(&ec, nodename)) < 0)
  fprintf(stderr, "ERROR: ei_connect failed");

使用 EPMD

注

erts:epmd 是 Erlang 端口映射守护进程。分布式 Erlang 节点在本地主机上向 epmd 注册，以向其他节点表明它们的存在并能接受连接。 epmd 维护一个节点和端口号信息的注册表，当一个节点希望连接到另一个节点时，它首先联系 epmd 以找到正确的连接端口号。

当您使用 ei_connect 连接到 Erlang 节点时，首先会连接到 epmd，如果节点已知，然后连接到 Erlang 节点。

C 节点也可以向 epmd 注册自己，以便系统中的其他节点能够找到并连接到它们。

在向 epmd 注册之前，您必须首先创建一个监听套接字并将其绑定到端口。然后：

int pub = ei_publish(&ec, port);

pub 是一个现在连接到 epmd 的文件描述符。epmd 监控连接的另一端。如果它检测到连接已被关闭，节点将取消注册。因此，如果您显式关闭描述符或您的节点失败，它将从 epmd 中取消注册。

注意

在某些系统上，由于操作系统不会自动关闭节点失败时留下的打开的描述符，因此这种机制无法检测到失败的节点。如果一个节点以这种方式失败，epmd 会阻止您使用旧名称注册新节点，因为它认为旧名称仍在使用中。在这种情况下，您必须显式关闭端口。

发送和接收 Erlang 消息

注

与 Erlang 一样，消息可以发送到一个 pid 或者一个已注册的名称。

发送消息

  • ei_send
  • ei_reg_send
  • 发送消息示例 {Pid, hello_world} 被发送到一个已注册的进程 my_server
ei_x_buff buf;
ei_x_new_with_version(&buf);

ei_x_encode_tuple_header(&buf, 2);
ei_x_encode_pid(&buf, ei_self(ec));
ei_x_encode_atom(&buf, "Hello world");

ei_reg_send(&ec, fd, "my_server", buf.buff, buf.index);

元组发送的第一个元素是您的自身进程 ID。这使 my_server 能够回复

  • 接收消息示例 {Pid, Something} 被接收。
erlang_msg msg;
int index = 0;
int version;
int arity = 0;
erlang_pid pid;
ei_x_buff buf;
ei_x_new(&buf);
for (;;) {
  int got = ei_xreceive_msg(fd, &msg, &x);
  if (got == ERL_TICK)
    continue;
  if (got == ERL_ERROR) {
    fprintf(stderr, "ei_xreceive_msg, got==%d", got);
    exit(1);
  }
  break;
}
ei_decode_version(buf.buff, &index, &version);
ei_decode_tuple_header(buf.buff, &index, &arity);
if (arity != 2) {
  fprintf(stderr, "got wrong message");
  exit(1);
}
ei_decode_pid(buf.buff, &index, &pid);

提示

为了提供健壮性，分布式 Erlang 节点偶尔会轮询其所有 连接邻居以检测失败的节点或通信链路。 一个接收此类消息的节点应立即响应，以一个 ERL_TICK 消息。这是由 ei_xreceive_msg() 自动完成的。然而，当这种情况发生时，ei_xreceive_msg 会将 ERL_TICK 返回给调用者，而不会将消息存储到 erlang_msg 结构中。

当收到一条消息时，调用者负责释放该接收到的消息。

远程过程调用

一个作为客户端的 Erlang 节点通常发送一个请求并等待回复。这种请求包含在远程节点的函数调用中，称为远程过程调用。

以下示例检查一个特定的 Erlang 进程是否存活

int index = 0, is_alive;
ei_x_buff args, result;

ei_x_new(&result);
ei_x_new(&args);
ei_x_encode_list_header(&args, 1);
ei_x_encode_pid(&args, &check_pid);
ei_x_encode_empty_list(&args);

if (ei_rpc(&ec, fd, "erlang", "is_process_alive",
           args.buff, args.index, &result) < 0)
    handle_error();

if (ei_decode_version(result.buff, &index) < 0
    || ei_decode_bool(result.buff, &index, &is_alive) < 0)
    handle_error();

使用全局名称

注

一个 C 节点可以通过 Kernel 中的 global 模块访问注册的名称。可以查找名称，允许 C 节点向命名的 Erlang 服务发送消息。C 节点还可以注册全局名称，允许它们为 Erlang 进程或其他 C 节点提供命名服务。

Erl_Interface 不提供全局服务的原生实现。相反，它使用由“附近”的 Erlang 节点提供的全局服务。

查看有哪些名称：

char **names;
int count;
int i;

names = ei_global_names(&ec,fd,&count);

if (names)
  for (i=0; i<count; i++)
    printf("%s\n",names[i]);

free(names);

ei_global_names 分配并返回一个包含 Kernel 中 global 模块所知所有名称的缓冲区。count 被初始化以指示数组中的名称数量。在 names 中的字符串数组以一个 NULL 指针终止，因此不需要使用 count 来确定何时到达最后一个名称。

调用者负责释放数组。ei_global_names通过单个调用malloc()分配数组以及所有字符串，因此 只需调用free(names)即可。 要查找其中一个名称：

ETERM *pid;
char node[256];
erlang_pid the_pid;

if (ei_global_whereis(&ec,fd,"schedule",&the_pid,node) < 0)
   fprintf(stderr, "ei_global_whereis error\n");

如果 "schedule" 在 Kernel 模块的 global 模块中已知，则将 Erlang 进程 ID 写入 _pid。这个进程 ID 可以用来向调度服务发送消息。此外，node 被初始化为包含服务注册的节点名称，这样您只需通过传递变量到 ei_connect 即可建立连接。

在注册名称之前，您应该已经使用 epmd 注册了您的端口号。这并非绝对必要，但如果您忽略了这一步骤，那么希望与您的服务通信的其他节点将无法找到或连接到您的进程。

创建一个 Erlang 进程可以使用以与您的服务通信的名称：

ei_global_register(fd,servicename,ei_self(ec));

注册名称后，使用 ei_accept 等待传入的连接。 注销一个名称：

ei_global_unregister(&ec,fd,servicename);
