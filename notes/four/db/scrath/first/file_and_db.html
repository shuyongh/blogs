文件 vs. 数据库

约 1764 字大约 6 分钟

2025-12-09

本章展示了简单地将数据转储到文件的局限性以及数据库解决的问题。

1.1 将数据持久化到文件

假设你有一些数据需要持久化到文件；这是一种典型的做法：

int SaveData1(const char *path, const unsigned char *data, size_t data_len) {
    FILE *fp = fopen(path, "wb");
    if (fp == NULL) {
        return -1;
    }
    
    size_t written = fwrite(data, 1, data_len, fp);
    int result = (written == data_len) ? 0 : -1;
    
    if (fclose(fp) != 0 && result == 0) {
        result = -1;
    }
    
    return result;
}

这种简单的方法有一些缺点：

  1. 它在更新文件之前截断了文件 。如果需要并发读取文件怎么办？
  1. 写入数据到文件可能不是原子 的，取决于写入的大小。并发读取器可能会得到不完整的数据。
  1. 数据何时实际持久化到磁盘 ？在 write 系统调用返回后，数据可能仍在操作系统的页面缓存中。系统崩溃并重新启动时，文件的状态是什么？

1.2 原子重命名

为了解决其中一些问题，我们提出一个更好的方法：

int SaveData2(const char *path, const unsigned char *data, size_t data_len) {
    char tmp_path[1024];
    
    //用当前时间和进程ID的组合作为种子，初始化随机数生成器
    //相同的种子会产生相同的随机数序列
    // 生成随机临时文件名
    srand(time(NULL) ^ getpid());
    int random_num = rand();
    //安全的格式化字符串
    snprintf(tmp_path, sizeof(tmp_path), "%s.tmp.%d", path, random_num);
    
    // 创建临时文件
    FILE *fp = fopen(tmp_path, "wb");
    if (fp == NULL) {
        return -1;
    }
    
    // 写入数据,数据先到缓冲区,此时数据还在C库的缓冲区，没到操作系统
    size_t written = fwrite(data, 1, data_len, fp);
    if (written != data_len) {
        fclose(fp);
        remove(tmp_path);
        return -1;
    }
    
    fclose(fp);
    
    // 重命名,原子操作
    if (rename(tmp_path, path) != 0) {
        remove(tmp_path);
        return -1;
    }
    
    return 0;
}

这种方法稍微复杂一些，它首先将数据转储到一个临时文件，然后将临时文件重命名为目标文件。这似乎避免了直接更新文件的非原子性问题——重命名操作是原子的。如果在重命名之前系统崩溃，原始文件保持不变，应用程序并发读取文件没有问题。

然而，这仍然有问题 ，因为它不控制数据何时持久化到磁盘，并且元数据（文件大小）可能在数据之前持久化到磁盘，可能在系统崩溃后损坏文件。（你可能已经注意到，一些日志文件在断电后包含零，这是文件损坏的迹象。）

1.3 fsync

要解决这个问题，我们必须在重命名之前将数据刷新到磁盘 。Linux 系统调用是 “fsync”。

int SaveData3(const char *path, const unsigned char *data, size_t data_len) {
    // 代码省略...
    // 同步数据到磁盘（关键！）
    // 强制写入操作系统,现在数据到了操作系统页面缓存
    if (fflush(fp) != 0) {
        fclose(fp);
        remove(tmp_path);
        return -1;
    }
    
    #ifdef __linux__
    // 使用 fsync 确保数据持久化,强制写入磁盘
    if (fsync(fileno(fp)) != 0) {
        fclose(fp);
        remove(tmp_path);
        return -1;
    }
    #endif
    
    fclose(fp);
    
    // 原子重命名
    if (rename(tmp_path, path) != 0) {
        remove(tmp_path);
        return -1;
    }
    
    return 0;
}

我们完成了吗？答案是否定的。我们已经将数据刷新到磁盘，但是元数据呢？我们是否也应该在包含文件的目录上调用 fsync？

这个兔子洞很深，这就是为什么在将数据持久化到磁盘时，数据库比文件更受青睐。

1.4 仅追加日志

在某些用例中，使用仅追加日志来持久化数据是有意义的。

// 创建日志
int LogCreate(const char *path) {
    //  是 Linux/Unix 系统调用，用于打开或创建文件。
    return open(path, O_RDWR | O_CREAT, 0664);
}

// 追加日志
int LogAppend(int fd, const char *line) {
    size_t line_len = strlen(line);
    size_t total_len = line_len + 1;
    
    char *buf = malloc(total_len);
    if (buf == NULL) {
        return -1;
    }
    
    memcpy(buf, line, line_len);
    buf[line_len] = '\n';
    
    ssize_t written = write(fd, buf, total_len);
    free(buf);
    
    if (written != total_len) {
        return -1;
    }
    
    if (fsync(fd) != 0) {
        return -1;
    }
    
    return 0;
}

仅追加日志的好处是它不修改现有数据，也不处理重命名操作，使其更能抵抗损坏。但仅靠日志不足以构建数据库。

  1. 数据库使用额外的"索引"来高效查询数据。对于任意顺序的记录，只有蛮力方法可以查询。
  2. 日志如何处理已删除的数据？它们不能永远增长。
