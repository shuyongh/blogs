并发读写

约 2557 字大约 9 分钟

12.1 读写者问题

为了支持并发请求，我们可以首先将事务分为只读事务和读写事务。仅读取者始终可以并发运行，因为它们不修改数据。而写入者必须修改树，必须被序列化（至少部分地）。

我们可以支持不同程度的并发。我们可以使用读写锁（RWLock）。它允许执行以下任一操作：

  • 多个并发读取者。
  • 单个写入者。

RWLock 是一种实用技术，易于添加。然而，由于使用了不可变数据结构，我们可以轻松实现更大程度的并发。

使用 RWLock，读取者可能被写入者阻塞，反之亦然。但使用不可变数据结构，写入者创建数据的新版本而不是覆盖当前版本，这允许读取者和写入者之间的并发，这优于 RWLock。这就是我们将要实现的。

我们的数据库将支持：

  • 多个并发读取者。
  • 多个读取者和单个写入者之间的并发。
  • 写入者完全序列化。

注意，通过仅部分序列化写入者，可以实现更大程度的并发。例如，如果读写事务读取数据的子集，然后使用该数据确定要写入的内容，我们可能能够并发执行读取操作，并仅序列化最终的提交操作。然而，这引入了新的问题：即使提交操作被序列化，写入者可能提交冲突的提交，因此我们需要额外的机制来防止或检测冲突。我们不会在本书中这样做。

12.2 分析实现

3 个主要变化：

首先，我们将事务类型拆分为两个，分别用于只读和读写事务。B-树类型被移动到事务类型（快照）。一个事务的读取不会受其他事务影响。

    mu     sync.Mutex
    writer sync.Mutex
    // 省略...
}

最后，空闲列表需要重新设计，因为我们不能重用仍然可以被读取者访问的页面。我们的解决方案是为每个版本的 B-树分配一个自动递增的版本号，当页面被释放时，空闲列表将版本号与页面号一起存储。只有版本号小于所有当前读取者的空闲页面才能被重用。

12.3 并发事务

第 1 部分：修改 KV 类型

  1. B-树类型被移动到事务类型，这里只保留根指针。同样，空闲列表也被移动。
  2. 管理磁盘页面的数据结构和代码也被移动到事务类型。
  3. 添加了互斥锁。写入者互斥锁用于序列化写入者，mu 互斥锁用于保护数据字段。
  4. 添加了版本号。以及一个正在进行的读取者列表，用于跟踪最小活动版本（用于空闲列表）。读取者列表作为堆数据结构维护，以便最小版本是第一个元素。
type KV struct {
    Path string
    // 内部
    fp *os.File

    // 修改 1：移动了 B-树和空闲列表
    tree struct {
        root uint64
    }
    free FreeListData

    mmap struct {
        // 相同；省略...
    }

    // 修改 2：移动了页面管理
    page struct {
        flushed uint64 // 数据库大小（页面数）
    }

    // 修改 3：互斥锁
    mu     sync.Mutex
    writer sync.Mutex

    // 修改 4：版本号和读取者列表
    version uint64
    readers ReaderList // 堆，用于跟踪最小读取者版本
}

第 2 部分：添加只读事务类型

// 只读 KV 事务
type KVReader struct {
    // 快照
    version uint64
    tree    BTree
    mmap struct {
        chunks [][]byte // 从 struct KV 复制。只读。
    }
    // 用于从堆中移除
    index int
}

B-树类型被移动到 KVReader 中，页面管理函数 pageGetMapped 也是如此。version 和 index 字段用于 ReaderList 堆。我们还复制了 mmap 块，因为它会被写入者修改。

func (kv *KV) BeginRead(tx *KVReader) {
    kv.mu.Lock()
    tx.mmap.chunks = kv.mmap.chunks
    tx.tree.root = kv.tree.root
    tx.tree.get = tx.pageGetMapped
    tx.version = kv.version
    heap.Push(&kv.readers, tx)
    kv.mu.Unlock()
}

func (kv *KV) EndRead(tx *KVReader) {
    kv.mu.Lock()
    heap.Remove(&kv.readers, tx.index)
    kv.mu.Unlock()
}

// BTree & FreeList 的回调，解引用指针。
func (tx *KVReader) pageGetMapped(ptr uint64) BNode

第 3 部分：添加读写事务类型

KVTX 扩展了 KVReader，因此它获得所有读取方法。与 B-树类型一样，空闲列表和页面管理数据结构也从 KV 类型移动。

// KV 事务
type KVTX struct {
    KVReader
    db   *KV
    free FreeList
    page struct {
        nappend int // 要追加的页面数
        // 新分配或释放的页面，键为指针。
        // nil 值表示已释放的页面。
        updates map[uint64][]byte
    }
}

要开始读写事务，必须获取写入者锁。然后我们可以初始化 B-树类型和空闲列表类型。不需要额外的锁来访问 KV 类型中的字段，因为写入者是唯一可以修改任何内容的线程。除了读取者列表（由读取者修改）。

tx.page* 函数只是从 KV 类型中移出。

// 开始事务
func (kv *KV) Begin(tx *KVTX) {
    tx.db = kv
    tx.page.updates = map[uint64][]byte{}
    tx.mmap.chunks = kv.mmap.chunks

    kv.writer.Lock()
    tx.version = kv.version

    // btree
    tx.tree.root = kv.tree.root
    tx.tree.get = tx.pageGet
    tx.tree.new = tx.pageNew
    tx.tree.del = tx.pageDel

    // freelist
    tx.free.FreeListData = kv.free
    tx.free.version = kv.version
    tx.free.get = tx.pageGet
    tx.free.new = tx.pageAppend
    tx.free.use = tx.pageUse
    tx.free.minReader = kv.version

    kv.mu.Lock()
    if len(kv.readers) > 0 {
        tx.free.minReader = kv.readers[0].version
    }
    kv.mu.Unlock()
}

回滚事务现在是无操作的，因为在提交之前 KV 类型中的任何内容都没有被修改。

// 结束事务：回滚
func (kv *KV) Abort(tx *KVTX) {
    kv.writer.Unlock()
}

只有在提交事务时才修改 KV 类型（对于树根和空闲列表头）。

// 结束事务：提交更新
func (kv *KV) Commit(tx *KVTX) error {
    defer kv.writer.Unlock()

    // 阶段 1：将页面数据持久化到磁盘。
    // 省略...

    // 此时事务可见。
    // 保存内存数据结构的新版本。
    kv.page.flushed += uint64(tx.page.nappend)
    kv.free = tx.free.FreeListData

    kv.mu.Lock()
    kv.tree.root = tx.tree.root
    kv.version++
    kv.mu.Unlock()

    // 阶段 2：更新主页面以指向新树。
    // 省略...
}

两个函数最后都释放写入者锁，显然。

12.4 空闲列表

空闲列表从 FILO（后进先出）改为 FIFO（先进先出）；新版本释放的页面被添加到列表头部，重用的页面从列表尾部移除。这使空闲列表保持排序（按版本号）。

为了避免重用仍然被读取者读取的页面，重用的页面必须来自不新于任何读取者的版本。这就是为什么我们将空闲列表设计为按版本号排序。

不幸的是，切换到 FIFO 引入了一些复杂性。我们必须遍历列表以访问列表的另一端（尾部）。我们将在内存中保留对每个节点的引用，这样我们只需要遍历列表一次。

// 由事务更新和提交的内存数据结构
type FreeListData struct {
    head uint64
    // 用于访问两端的缓存指针到列表节点。
    nodes []uint64 // 从尾部到头部
    // 缓存的项目总数；存储在头节点中。
    total int
    // 缓存尾部节点中已丢弃的项目数。
    offset int
}

type FreeList struct {
    FreeListData
    // 对于每个事务
    version   uint64 // 当前版本
    minReader uint64 // 最小读取者版本
    freed     []uint64 // 将添加到空闲列表的页面

    // 用于管理磁盘页面的回调
    get func(uint64) BNode // 解引用指针
    new func(BNode) uint64 // 追加新页面
    use func(uint64, BNode) // 重用页面
}

版本号被添加到空闲列表节点和主页面：

| 类型 | 大小 | 总数 | 下一个 | 指针-版本-对 |
| 2B  | 2B  | 8B  | 8B  | 大小 * 16B |

主页面格式：

| 签名 | btree_root | page_used | free_list | 版本 |
| 16B  | 8B         | 8B        | 8B        | 8B   |

所有读取者的最旧版本在事务开始时获取。

// 开始事务
func (kv *KV) Begin(tx *KVTX) {
    // 省略...
    tx.free.minReader = kv.version
    kv.mu.Lock()
    if len(kv.readers) > 0 {
        tx.free.minReader = kv.readers[0].version
    }
    kv.mu.Unlock()
}

当从列表尾部重用页面时检查它。

// 尝试从尾部移除一个项目。失败时返回 0。
// 移除的指针必须不能被最小版本读取者访问。
func (fl *FreeList) Pop() uint64 {
    fl.loadCache()
    return flPop1(fl)
}

func flPop1(fl *FreeList) uint64 {
    if fl.total == 0 {
        return 0
    }

    // 从尾部移除一个项目
    assert(fl.offset < flnSize(fl.get(fl.nodes[0])))
    ptr, ver := flnItem(fl.get(fl.nodes[0]), fl.offset)
    if versionBefore(fl.minReader, ver) {
        // 不能使用；可能被最小版本读取者访问。
        return 0
    }

    fl.offset++
    fl.total--

    // 丢弃空节点并移动到下一个节点
    // 省略...
    return ptr
}

// a < b
func versionBefore(a, b uint64) bool {
    return int64(a-b) < 0
}

12.5 结束语

我们已经展示了不可变数据结构的一个巨大优势，即读取者和写入者之间的并发很容易。只读事务可以运行任意长的时间，唯一的缺点是长时间运行的读取者会阻止页面重用。然而，读写事务预计会很短，因为写入者完全序列化。这种程度的并发对于某些用例可能已经足够。

我们已经在本书中探讨了三个主要主题：持久化、索引和并发现在是时候为我们的数据库添加一个用户界面了——一种查询语言——这是下一章的主题。
