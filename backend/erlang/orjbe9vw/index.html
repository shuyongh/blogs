路由

约 1889 字大约 6 分钟

2025-03-28

牛仔默认不做任何事情。

为了让 Cowboy 有用，你需要将 URI 映射到将处理请求的 Erlang 模块。这被称为路由。

Cowboy 使用以下算法路由请求：

  • 如果没有配置的主机匹配请求 URI，则返回 400 响应。
  • 否则，将使用第一个匹配请求 URI 的配置主机。仅考虑该主机配置的路径。
  • 如果前面步骤中配置的路径都不匹配请求 URI，则返回 404 响应。
  • 否则，将处理器及其初始状态添加到环境中，并继续处理请求。

注意：有可能出现两个主机都匹配请求 URI 的情况，但只有第二个主机上的路径匹配请求 URI。在这种情况下，预期结果是返回 404 响应，因为路由过程中仅使用第一个匹配请求 URI 的配置主机上的路径。

路由规则需要编译后才能被 Cowboy 使用。编译的结果是调度规则。

语法

路由的一般结构如下定义。

Routes = [Host1, Host2, ... HostN].

每个主机包含匹配主机的规则，以及可选的约束条件，以及路径组件的路由列表。

Host1 = {HostMatch, PathsList}.
Host2 = {HostMatch, Constraints, PathsList}.

路径组件的路由列表定义类似于主机列表。

PathsList = [Path1, Path2, ... PathN].

最后，每条路径包含与路径匹配的规则，还可以包含可选约束，并给出了要使用的处理模块及其初始状态。

  • 初始状态(``InitialState)将传递给当前路由的处理器中init/2的第二个参数
Path1 = {PathMatch, Handler, InitialState}.
Path2 = {PathMatch, Constraints, Handler, InitialState}.

继续阅读以了解匹配语法和可选约束的更多信息。

匹配语法

匹配语法用于将主机名和路径与其相应的处理模块关联起来。

路由规则的匹配语法与主机和路径相同，但有一些细微差别。实际上，段落分隔符是不同的，主机是从最后一个段落开始匹配到第一个段落。所有示例都将包含主机和路径匹配规则，并在遇到差异时进行解释。

除了我们在本节末尾解释的特殊值外，最简单的匹配值是主机或路径。它可以作为字符串() 或二进制() 给出。

PathMatch1 = "/".
PathMatch2 = "/path/to/resource".

HostMatch1 = "cowboy.example.org".

如您所见，以这种方式定义的所有路径都必须以斜杠字符开头。请注意，这两个路径从路由的角度来看是相同的。

PathMatch2 = "/path/to/resource".
PathMatch3 = "/path/to/resource/".

带有和不带尾随点的主机对于路由来说是等价的。同样，带有和不带前导点的主机也是等价的。

HostMatch1 = "cowboy.example.org".
HostMatch2 = "cowboy.example.org.".
HostMatch3 = ".cowboy.example.org".

可以提取主机和路径的部分段，并将这些值存储在 Req 对象中以供以后使用。我们称这些类型的值为绑定。

绑定的语法非常简单。一个以 : 字符开始的段意味着该段中的内容直到段结束都是绑定名称，该段的值将被存储在该绑定名称中。

PathMatch = "/hats/:name/prices".
HostMatch = ":subdomain.example.org".

如果在路由时这些匹配成功，你将得到两个绑定，subdomain 和 name，每个绑定包含它们定义的段值。例如，URL http://test.example.org/hats/wild_cowboy_legendary/prices 将导致将值 test 绑定到名称 subdomain，并将值 wild_cowboy_legendary 绑定到名称 name。稍后可以使用 cowboy_req:binding/{2,3} 获取这些绑定。绑定名称必须以原子形式给出。

你可以使用一个特殊的绑定名称来模拟 Erlang 中的下划线变量。任何与 _ 绑定匹配都将成功，但数据将被丢弃。这对于一次匹配多个域名特别有用。

HostMatch = "ninenines.:_".

同样，可以有可选段。方括号内的内容是可选的。

PathMatch = "/hats/[page/:number]".
HostMatch = "[www.]ninenines.eu".

你也可以嵌套可选段。

PathMatch = "/hats/[page/[:number]]".

尽管 Cowboy 不会拒绝在路由中使用多个方括号，但如果路由不明确，则行为可能是未定义的。例如，这个路由需要约束来确定什么是章节，什么是页面，因为它们都是可选的：

PathMatch = "/book/[:chapter]/[:page]".

你可以使用 [...] 获取主机或路径的剩余部分。对于主机，它将匹配之前的内容；对于路径，它将匹配之前匹配的段落之后的内容。这是一个可选段的特殊情况，因为它可以有零个、一个或多个段。然后你可以使用 cowboy_req:host_info/1 和 cowboy_req:path_info/1 分别找到这些段。它们将被表示为段的列表。

PathMatch = "/hats/[...]".
HostMatch = "[...]ninenines.eu".

如果路由规则中出现两次绑定，那么匹配只有在它们具有相同值时才会成功。这复制了 Erlang 的模式匹配行为。

PathMatch = "/hats/:name/:name".

当可选段存在时，这一点也同样适用。在这种情况下，只有当该段可用时，两个值才必须相同。

PathMatch = "/hats/:name/[:name]".

如果绑定同时定义在主机和路径中，那么它们也必须具有相同的值。

PathMatch = "/:user/[...]".
HostMatch = ":user.github.com".

最后，有两种特殊的匹配值可以使用。第一个是原子 '_'，它可以匹配任何主机或路径。

PathMatch = '_'.
HostMatch = '_'.

第二个是特殊的主机匹配 "*"，它将匹配通配符路径，通常与 OPTIONS 方法一起使用。

HostMatch = "*".

约束

匹配完成后，可以将生成的绑定与一组约束进行测试。只有在绑定定义时才会测试约束。它们将按照你定义的顺序运行。只有当它们全部成功时，匹配才会成功。如果匹配失败，Cowboy 将尝试列表中的下一个路由。

约束的格式与 cowboy_req 中的匹配函数相同：它们以包含一个或多个约束的字段列表的形式提供。虽然路由器接受相同的格式，但它会跳过没有约束的字段，并且还会忽略任何默认值。

了解更多关于约束的内容。

编译

路由必须在 Cowboy 使用它们之前进行编译。编译步骤将路由规范化以简化代码并加快执行速度，但最终仍然逐个查找路由。 可以采用更快速的编译策略直接将路由编译为 Erlang 代码，但这样会需要更重的依赖。

要编译路由，只需调用相应的函数：

Dispatch = cowboy_router:compile([
    %% {HostMatch, list({PathMatch, Handler, InitialState})}
    {'_', [{'_', my_handler, #{}}]}
]),
%% Name, TransOpts, ProtoOpts
cowboy:start_clear(my_http_listener,
    [{port, 8080}],
    #{env => #{dispatch => Dispatch}}
).

使用 persistent_term

路由可以从 Erlang/OTP 21.2 的 persistent_term 中存储。当路由数量很大时，这可能会提高性能。

要使用此功能，您需要编译路由，将它们存储在 persistent_term 中，然后通知 Cowboy：

Dispatch = cowboy_router:compile([
    {'_', [{'_', my_handler, #{}}]}
]),
persistent_term:put(my_app_dispatch, Dispatch),
cowboy:start_clear(my_http_listener,
    [{port, 8080}],
    #{env => #{dispatch => {persistent_term, my_app_dispatch}}}
).

实时更新

你可以使用 cowboy:set_env/3 函数来更新由路由使用的分发列表。这将应用于监听器接受的所有新连接：

Dispatch = cowboy_router:compile(Routes),
cowboy:set_env(my_http_listener, dispatch, Dispatch).

请注意，在更新之前需要重新编译路由。

当使用 persistent_term 时，无需调用此函数，你可以直接将新的路由放入存储中。
