监督树

Erlang/OTP 中的一个基本概念是 监督树 。这是一个过程 基于 工作者 和 监督者 思想的结构化模型：

  • 工作者是执行计算和其他实际工作的进程。
  • 监督者是监视工作程序的进程。如果出现问题，监督者可以重新启动一个工作者。
  • 监督树是将代码分层排列成监督者和工作者的结构，这使得设计和编程容错软件成为可能

行为

在监督树中，许多进程具有类似的结构并遵循类似的模式。例如，监督者共享类似的结构，唯一的区别在于它们所监督的子进程。许多工作进程是服务器-客户端关系中的服务器、有限状态机或事件处理程序。

行为是这些常见模式的形式化。这个想法是将流程的代码分为通用部分（行为模块）和特定部分（回调模块）

behaviour 模块是 Erlang/OTP 的一部分。要实现 supervisor 等流程，用户只需要实现回调模块，即导出一组预定义的函数，即回调函数

标准的 Erlang/OTP 行为是：

  • gen_server

    用于实现客户端-服务器关系的服务器

  • gen_statem

    用于实现状态机

  • gen_event

    用于实现事件处理功能

  • supervisor

    用于在监督树中实施监督器

不使用behaviour示例

不使用 behaviour 编写的代码可以更高效，但效率的提高是以牺牲通用性为代价的。以一致的方式管理系统中的所有应用程序的能力非常重要。

使用 behaviours 还可以更轻松地阅读和理解其他程序员编写的代码。临时编程结构虽然可能更有效，但总是更难理解。

下面的示例说明了如何将代码划分为泛型部分和特定部分。考虑以下代码（用纯 Erlang 编写）用于一个简单的服务器，它跟踪许多 “通道” 。其他进程可以通过分别调用函数 alloc/0 和 free/1 来分配和释放通道

请注意以下事项：

  • server 中的代码可以重用来构建许多不同的服务器。
  • 服务器名称（在本例中为 atom ch2）对 Client 端函数的用户是隐藏的。这意味着可以更改名称而不会影响它们。
  • 协议 （发送到服务器和从服务器接收的消息） 也被隐藏。这是很好的编程实践，它允许人们使用接口函数在不更改代码的情况下更改协议。
  • 无需更改 ch2 即可扩展 server 的功能 或任何其他回调模块。
服务器代码

对应于 Erlang/OTP 行为的 gen_server。

-module(server).
-export([start/1]).
-export([call/2, cast/2]).
-export([init/1]).

start(Mod) ->
    spawn(server, init, [Mod]).

call(Name, Req) ->
    Name ! {call, self(), Req},
    receive
        {Name, Res} ->
            Res
    end.

cast(Name, Req) ->
    Name ! {cast, Req},
    ok.

init(Mod) ->
    register(Mod, self()),
    State = Mod:init(),
    loop(Mod, State).

loop(Mod, State) ->
    receive
        {call, From, Req} ->
            {Res, State2} = Mod:handle_call(Req, State),
            From ! {Mod, Res},
            loop(Mod, State2);
        {cast, Req} ->
            State2 = Mod:handle_cast(Req, State),
            loop(Mod, State2)
    end.
回调模块
-module(ch2).
-export([start/0]).
-export([alloc/0, free/1]).
-export([init/0, handle_call/2, handle_cast/2]).

start() ->
    server:start(ch2).

alloc() ->
    server:call(ch2, alloc).

free(Ch) ->
    server:cast(ch2, {free, Ch}).

init() ->
    channels().

handle_call(alloc, Chs) ->
    alloc(Chs). % => {Ch,Chs2}

handle_cast({free, Ch}, Chs) ->
    free(Ch, Chs). % => Chs2


channels() ->
   {_Allocated = [], _Free = lists:seq(1, 100)}.

alloc({Allocated, [H|T] = _Free}) ->
   {H, {[H|Allocated], T}}.

free(Ch, {Alloc, Free} = Channels) ->
   case lists:member(Ch, Alloc) of
      true ->
         {lists:delete(Ch, Alloc), [Ch|Free]};
      false ->
         Channels
   end.

应用

Erlang/OTP 带有许多组件，每个组件都实现了一些特定的功能。组件使用Erlang/OTP术语称为应用程序。Erlang/OTP 应用程序的示例包括 Mnesia，它具有对数据库服务进行编程所需的一切，以及用于调试 Erlang 程序的 Debugger。

基于 Erlang/OTP 的最小系统由以下两个应用程序组成：

  • 内核 - 运行 Erlang 所需的功能
  • STDLIB - Erlang 标准库

应用程序概念适用于程序结构 （进程） 和目录结构 （模块）。

最简单的应用程序没有任何进程，而是由一组功能模块组成。这样的应用程序称为库应用程序。库应用程序的一个示例是 STDLIB。

使用标准行为将带有进程的应用程序最容易实现为监督树。

发行

发行版是由 Erlang/OTP 应用程序的子集和一组特定于用户的应用程序组成的完整系统。

如何对版本进行编程，请参见 版本。

如何在目标环境中安装发行版 在系统原则中创建和升级目标系统。
