事件处理原则

在 OTP 中，事件管理器是一个可以发送事件的命名对象。事件可以是错误、警报或需要记录的信息。

在事件管理器中，可以安装零个、一个或多个事件处理器。当事件管理器被通知有关事件时，该事件由所有已安装的事件处理器进行处理。例如，用于处理错误的默认事件管理器可以安装一个处理器，该处理器将错误消息写入终端。如果要在一定时期内将错误消息保存到文件中，用户可以添加另一个执行此操作的事件处理器。当不再需要将日志记录到文件时，删除此事件处理器

事件管理器作为进程实现，每个事件处理器作为回调模块实现

事件管理器本质上维护一个 {Module, State} 对列表，其中每个 Module 是一个事件处理器， State 是该事件处理器的内部状态。

示例

事件处理器将错误消息写入终端的回调模块可以如下所示：

-module(terminal_logger).
-behaviour(gen_event).

-export([init/1, handle_event/2, terminate/2]).

init(_Args) ->
    {ok, []}.

handle_event(ErrorMsg, State) ->
    io:format("***Error*** ~p~n", [ErrorMsg]),
    {ok, State}.

terminate(_Args, _State) ->
    ok.

事件处理器将错误消息写入文件的回调模块可以如下所示：

-module(file_logger).
-behaviour(gen_event).

-export([init/1, handle_event/2, terminate/2]).

init(File) ->
    {ok, Fd} = file:open(File, read),
    {ok, Fd}.

handle_event(ErrorMsg, Fd) ->
    io:format(Fd, "***Error*** ~p~n", [ErrorMsg]),
    {ok, Fd}.

terminate(_Args, Fd) ->
    file:close(Fd).

代码在下一节中解释。

开始一个事件管理器

要启动一个用于处理错误的的事件管理器，如前一个示例中所述，请调用以下函数：

gen_event:start_link({local, error_man})

gen_event:start_link/1 启动并链接到一个新的事件管理器进程。

参数{local, error_man}指定了事件管理器应在本地注册的名称。该参数也可以作为{global, Name}给出，以全局注册事件管理器。 全局：register_name/2.

如果省略了名称，则事件管理器未注册。相反，必须使用其进程 ID。

如果事件管理器是监督树的一部分，意味着它是由监督器启动的，则必须使用gen_event:start_link/1。还有一个函数gen_event:start/1，用于启动不属于监督树的自立式事件管理器。

添加事件处理器

以下示例展示了如何通过使用 shell 启动事件管理器并向其中添加事件处理器：

1> gen_event:start({local, error_man}).
{ok,<0.31.0>}
2> gen_event:add_handler(error_man, terminal_logger, []).
ok

此函数向注册为error_man的事件管理器发送消息，告知其添加事件处理器terminal_logger。事件管理器调用回调函数terminal_logger:init([])，其中参数[]是add_handler的第三个参数。init/1预期返回{ok, State}，其中State是事件处理器的内部状态。

init(_Args) ->
    {ok, []}.

这里，init/1不需要任何输入数据，并忽略其参数。 终端记录器的内部状态未使用。对于文件记录器，内部状态用于保存打开的文件描述符。

init(File) ->
    {ok, Fd} = file:open(File, read),
    {ok, Fd}.

通知事件

3> gen_event:notify(error_man, no_reply).
***Error*** no_reply
ok

错误处理程序是事件管理器的名称，而 error_man 是事件。

事件被转换为消息并发送给事件管理器。当接收到事件时，事件管理器按照它们添加的顺序，为每个已安装的事件处理器调用handle_event(Event, State)。该函数预期返回一个元组{ok,State1}，其中State1是事件处理器状态的新值。

在 终端记录器：

handle_event(ErrorMsg, State) ->
    io:format("***Error*** ~p~n", [ErrorMsg]),
    {ok, State}.copy

在file_logger：

handle_event(ErrorMsg, Fd) ->
    io:format(Fd, "***Error*** ~p~n", [ErrorMsg]),
    {ok, Fd}.copy

删除事件处理器

4> gen_event:delete_handler(error_man, terminal_logger, []).
okcopy

此函数向注册为 error_man 的事件管理器发送消息，告知其删除事件处理程序 terminal_logger。事件管理器调用回调函数 terminal_logger:terminate([], State) ，其中参数 [] 是 delete_handler 的第三个参数。terminate/2 应与 init/1 相反，并执行任何必要的清理。其返回值被忽略。

对于terminal_logger，无需清理：

terminate(_Args, _State) ->
    ok.

对于file_logger，在init中打开的文件描述符必须关闭：

terminate(_Args, Fd) ->
    file:close(Fd).

停止

当事件管理器停止时，它会给每个已安装的事件处理器机会通过调用terminate/2来清理，就像删除处理器时一样。

在一个监督树

如果事件管理器是监督树的一部分，则不需要停止函数。事件管理器将由其监督者自动终止。具体如何执行由监督者中设置的关闭策略定义。

独立事件管理器

事件管理器也可以通过调用以下方式停止：

1> gen_event:stop(error_man).
ok

处理其他消息

如果gen_event进程需要能够接收除事件之外的其他消息，则必须实现回调函数handle_info(Info, State)来处理它们。其他消息的例子包括，如果事件管理器连接到除监督者之外的其他进程（例如通过gen_event:add_sup_handler/3），并且正在捕获退出信号，则退出消息。

handle_info({'EXIT', Pid, Reason}, State) ->
    %% Code to handle exits here.
    ...
    {noreply, State1}.

要实现的最后一个函数是 code_change/3：

code_change(OldVsn, State, Extra) ->
    %% Code to convert state (and more) during code change.
    ...
    {ok, NewState}.
