应用概念

在实现特定功能的代码创建完成后，您可能需要将其转换为一个应用程序——一个可以作为一个单元启动和停止，并且可以在其他系统中重用的组件。

创建应用程序的步骤如下：

  • 创建一个应用程序回调模块，描述如何启动和停止应用程序。
  • 创建一个应用程序规范，并将其放在应用程序资源文件中。此文件会指定应用程序包含哪些模块以及回调模块的名称等其他内容。

如果使用 systools ，Erlang/OTP 的打包代码工具（参见 Releases），每个应用的代码将被放置在按照预定义目录结构分开的目录中。

常用方法

签名说明示例
application:start(Application)启动应用程序
application:which_applications()查看当前运行的所有应用程序
application:get_env(Application,Key)查看应用程序资源文件中的env
application:load(Application)加载应用程序
application:unload(Application).卸载应用程序
application:ensure_all_started(Application)启动应用程序及其所有依赖项的标准方法
依赖自动解析： 自动按照 .app 文件中的 applications 列表启动所有依赖 正确处理依赖的启动顺序
幂等操作： 如果应用已启动，不会重复启动 如果部分依赖已启动，只会启动未启动的部分
事务性启动： 任一依赖启动失败时，会自动停止已启动的应用 避免系统处于不一致状态

应用回调模块

如何启动和停止应用程序的代码（包括其监督树）是由两个回调函数描述的：

start(StartType, StartArgs) -> {ok, Pid} | {ok, Pid, State}
stop(State)

  • start/2 在启动应用程序时被调用，用于启动顶级监督者并创建监督树。它应该返回顶级监督者的进程标识符和一个可选项 State ，默认值为 [] 。这个项会原封不动地传递给 stop/1 。

    • StartType 通常是原子值 normal 。只有在接管或故障转移的情况下，它才具有其他值；请参见分布式应用程序。
    • StartArgs 由应用资源文件中的键 mod 定义。

  • stop/1 在应用程序停止后调用，用于执行任何必要的清理工作。实际上停止应用程序，即关闭监督树，会自动处理，如在启动和停止应用程序中所述。

监督行为中打包监督树的应用回调模块示例：

-module(application_app).
-behaviour(application).

-export([start/2, stop/1]).

start(_Type, _Args) ->
    application_sup:start_link().

stop(_State) ->
    ok.

一个无法启动或停止的库应用不需要任何应用回调模块。

应用程序资源文件

这个文件将告诉 Erlang VM 应用程序是什么，从哪里开始，在哪里结束。这个文件与所有编译后的模块一起存在于 ebin/ 目录中。

这个文件通常命名为 <yourapp>.app，并包含一堆 Erlang 项，这些项以 VM 可以理解的方式定义了应用程序

{application, test, [Opt1,...,OptN]}.
  • Application ，一个原子，是该应用的名称。文件必须命名为 test.app 。
  • 每个 Opt 是一个元组 {Key,Value} ，用于定义应用程序的某些属性。所有键都是可选的。省略的键将使用默认值。

一个库应用的最小 .app.src 文件内容如下：

{application, test, []}.

一个监督 树应用的最小 .app 文件Application.app内容如下：

{application, test,
 [{mod, {test_app,[]}}]}.

当使用 systools 时，需要指定 Erlang/OTP 工具用于打包代码的选项（参见章节 发行版），包括 description 、 vsn 、 modules 、 registered 和 applications

{application, test,
 [{description, "Channel allocator"},
  {vsn, "1"},
  {modules, [test_app, test_sup, ch3]},
  {registered, [ch3]},
  {applications, [kernel, stdlib, sasl]},
  {mod, {test_app,[]}}
 ]}.
  • description - 简短描述，字符串类型。默认值为 "" 。
  • vsn - 版本号，一个字符串。默认值为 "" 。
  • modules - 由该应用程序引入的所有模块。 systools 在生成启动脚本和 tar 文件时使用此列表。一个模块只能包含在一个应用程序中。默认值为 [] 。
  • registered - 应用程序中所有注册进程的名称。 systools 使用此列表来检测应用程序之间的名称冲突。默认值为 [] 。
  • applications - 在此应用启动之前必须启动的所有应用。 systools 使用此列表来生成正确的启动脚本。默认值为 [] 。请注意，所有应用至少都依赖于 Kernel 和 STDLIB。

提示

对于应用资源文件的语法和内容详情，请参见 Kernel 中的 app。

目录结构

当使用 systools 打包代码时，每个应用的代码会被放置在单独的目录 lib/Application-Vsn 中，其中 Vsn 是版本号。

这在即使不使用 systools 的情况下也很有用，因为 Erlang/OTP 是根据 OTP 原则打包的，因此带有特定的目录结构。代码服务器（参见 Kernel 模块中的 code ）会自动使用目录中版本号最高的代码，如果存在多个版本的应用程序的话。

开发环境的目录结构指南

任何开发目录结构都可以，只要发布的目录结构符合以下描述即可，但建议在开发环境中也使用相同的目录结构。由于版本号是发布步骤的产物，因此不应包含在应用程序目录名称中。

一些子目录是必需的。一些子目录是可选的，这意味着只有在应用程序本身需要时才使用。最后，一些子目录是推荐的，意味着鼓励使用，并且应按照此处所述使用。例如，为了被视为一个合格的 OTP 应用程序，应用程序中同时存在文档和测试是鼓励的。

    ─ ${application}
      ├── doc
      │   ├── internal
      │   ├── examples
      │   └── src
      ├── include
      ├── priv
      ├── src
      │   └── ${application}.app.src
      |   └── ${application}_app.erl
      |   └── ${application}_sup.erl
      └── test
  • src - 必需。包含 Erlang 源代码、 .app 文件的源代码以及应用程序本身使用的内部包含文件。 src 内可以使用子目录作为命名空间来组织源文件。这些目录的深度应不超过一层。
  • priv - 可选。保存可执行文件、其他程序以及应用程序正常工作所需的其他特定文件
  • include - 可选。用于公共包含文件，这些文件必须从其他应用程序中访问。
  • doc - 推荐。任何源文档应放置在此处的子目录中。
  • doc/internal - 推荐。任何描述此应用实现细节的文档（不打算出版）应放置在此处。
  • doc/examples - 推荐。应将此应用使用示例的源代码放置于此。鼓励将示例从该目录引用到公共文档。
  • doc/src - 推荐。所有用于文档的源文件，如 Markdown、AsciiDoc 或 XML 文件，都应放置在此处。
  • test - 推荐。所有与测试相关的文件，如测试套件和测试规范，应放置在此处。

其他开发环境中的目录可能也需要。如果使用了除 Erlang 以外的语言的源代码，例如用于 NIFs 的 C 代码，应将该代码放在一个单独的目录中。按照惯例，建议用语言名称前缀这些目录，例如 C 语言目录可以命名为 c_src ，Java 语言目录可以命名为 java_src ，Go 语言目录可以命名为 go_src 。以 _src 结尾的目录表示这是应用程序的一部分，并且是编译步骤的一部分。最终的构建产物应针对 priv/lib 或 priv/bin 目录。

priv 目录存放应用程序在运行时所需的资产。可执行文件应位于 priv/bin 中，动态链接库应位于 priv/lib 中。其他资产可以位于 priv 目录中，但建议以结构化的方式存放。

源代码来自其他语言（如 ASN.1 或 Mibs）并生成 Erlang 代码的文件，应放置在目录中，位于顶层目录或 src 中，并以源语言名称命名，例如 asn1 和 mibs 。构建产物应放置在其相应的语言目录中，例如 src 用于 Erlang 代码或 java_src 用于 Java 代码。

在开发环境中，如果发布的 .app 文件位于 ebin 目录中是可以接受的，但建议它是构建步骤的产物。按照惯例，位于 src 目录中的 .app.src 文件被使用。这个文件几乎与 .app 文件相同，但在构建步骤中会替换某些字段，例如应用程序版本。

目录名称不应大写。

建议省略空目录。

已发布系统的目录结构

一个发布的应用程序必须遵循一定的结构。

    ─ ${application}-${version}
      ├── bin
      ├── doc
      │   ├── html
      │   ├── man[1-9]
      │   ├── pdf
      │   ├── internal
      │   └── examples
      ├── ebin
      │   └── ${application}.app
      ├── include
      ├── priv
      │   ├── lib
      │   └── bin
      └── src
  • src - 可选。包含应用程序自身使用的 Erlang 源代码和内部 include 文件。
  • ebin - 必需。包含 Erlang 对象代码，即 .beam 文件。 .app 文件也必须放置在此处。
  • priv - 可选。用于应用程序特定的文件。 code:priv_dir/1 用于访问此目录。
  • priv/lib - 推荐。如果应用程序使用了任何共享对象文件，例如 NIF 或链接的驱动程序，应将这些文件放置在此处。
  • priv/bin - 推荐。任何由应用程序使用的可执行文件，例如端口程序，应放置在此处。
  • include - 可选。用于公共包含文件，这些文件必须从其他应用程序中访问。
  • bin - 可选。任何应用程序的可执行文件，如 escripts 或 shell 脚本，应放置在此处。
  • doc - 可选。任何已发布的文档应放在这里的子目录中。

src 目录在调试目的下可能会有用，但这是可选的。只有当应用程序包含公共头文件时，才应发布 include 目录。

建议省略空目录。

应用程序控制器

当启动一个 Erlang 运行时，系统会启动 Kernel 应用程序的一部分进程。其中一个进程是应用程序控制器过程，注册为 application_controller 。它启动所有其他应用程序，并位于大多数应用程序之上。事实上，你可以说应用程序控制器有点像所有应用程序的监督者

**应用程序控制器(AC)**使用 Kernel 中的模块 application 来加载、卸载、启动和停止应用程序。

提示

应用程序控制器在技术上并不位于所有应用程序之上。一个例外是内核应用程序，它本身启动一个名为user的进程。实际上，user进程充当应用程序控制器的组长，因此内核应用程序需要一些特殊处理。我们不必关心这一点，但我觉得为了精确起见应该包含它。

在 Erlang 中，IO 系统依赖于一个名为组长的概念。组长代表标准输入和输出，并被所有进程继承。有一个隐藏的IO 协议，组长和任何调用 IO 函数的进程都与之通信。然后，组长负责将这些消息转发到任何存在的输入/输出通道，并在我们本文的范围内做一些我们不关心的魔法。

无论如何，当有人决定要启动一个应用程序时，应用程序控制器（在 OTP 行话中通常简称为AC）会启动一个应用程序管理器。应用程序管理器实际上是两个负责每个独立应用程序的进程：它们设置应用程序并充当应用程序顶级监督者和应用程序控制器之间的中间人。

加载和卸载应用程序

在启动应用程序之前，必须先加载它。应用程序控制器会读取并存储来自 .app 文件的信息：

1> application:load(test).
ok
2> application:loaded_applications().
[{kernel,"ERTS  CXC 138 10","2.8.1.3"},
 {stdlib,"ERTS  CXC 138 10","1.11.4.3"},
 {ch_app,"Channel allocator","1"}]

已停止的应用程序或从未启动的应用程序可以被卸载。应用程序的相关信息将从应用程序控制器的内部数据库中删除。

3> application:unload(test).
ok
4> application:loaded_applications().
[{kernel,"ERTS  CXC 138 10","2.8.1.3"},
 {stdlib,"ERTS  CXC 138 10","1.11.4.3"}]

注意

加载/卸载应用程序不会加载/卸载应用程序使用的代码。代码加载由代码服务器以常规方式处理。

启动和停止应用程序

一个应用程序通过调用以下内容启动：

5> application:start(test).
ok
6> application:which_applications().
[{kernel,"ERTS  CXC 138 10","2.8.1.3"},
 {stdlib,"ERTS  CXC 138 10","1.11.4.3"},
 {ch_app,"Channel allocator","1"}]

如果该应用尚未加载，应用控制器将首先使用 application:load/1 加载它。它会检查 applications 键的值以确保在此应用之前需要启动的所有应用均已运行。

随后，应用控制器为该应用创建一个应用主控。

应用主节点将其自身确立为应用程序中所有进程的组领导者，并将转发 I/O 给之前的组领导者。

注意

应用主的目的作为组领导是为了容易跟踪属于该应用的哪些进程。这需要支持 application:get_application/0 和 application:get_env/1 函数，同时在停止应用时确保所有属于该应用的进程都被终止。

应用程序主模块通过调用模块中的start/2回调函数来启动应用程序，该函数的启动参数由 .app 文件中的 mod 键定义。

调用以下函数可停止但不卸载一个应用程序：

7> application:stop(test).

应用程序主节点停止应用的方式是告诉顶级监督者关闭。顶级监督者会告诉所有其子进程关闭，依此类推；整个树状结构将按照启动顺序的逆序终止。然后，应用主进程调用由 mod 键定义的模块中的 stop/1 应用回调函数。

配置一个应用程序

一个应用程序可以使用配置参数进行配置。这些参数是一个由键 env 指定的 {Par,Val} 元组列表：在 .app 文件中指定。

{application, test,
 [{description, "Channel allocator"},
  {vsn, "1"},
  {modules, [test_app, test_sup, ch3]},
  {registered, [ch3]},
  {applications, [kernel, stdlib, sasl]},
  {mod, {test_app,[]}},
  {env, [{file, "/usr/local/log"}]}
 ]}.

Par 应该是一个原子。 Val 是任意项。应用程序可以通过调用 application:get_env(App, Par) 或其他类似函数来检索配置参数的值。更多信息，请参见内核模块 application 。

示例:

% erl
Erlang (BEAM) emulator version 5.2.3.6 [hipe] [threads:0]

Eshell V5.2.3.6  (abort with ^G)
1> application:start(test).
ok
2> application:get_env(test, file).
{ok,"/usr/local/log"}copy

.app 文件中的值可以通过系统配置文件中的值覆盖。这是一个包含相关应用配置参数的文件：

[{Application1, [{Par11,Val11},...]},
 ...,
 {ApplicationN, [{ParN1,ValN1},...]}].

系统配置命名为 Name.config ，并使用命令行参数 -config Name 启动 Erlang。详情请参见 Kernel 中的 config 。

示例:

一个文件 test.config 包含以下内容创建：

[{ch_app, [{file, "testlog"}]}].

file 的值覆盖了 .app 文件中定义的 file 的值：

% erl -config test
Erlang (BEAM) emulator version 5.2.3.6 [hipe] [threads:0]

Eshell V5.2.3.6  (abort with ^G)
1> application:start(test).
ok
2> application:get_env(test, file).
{ok,"testlog"}copy

如果使用了发布处理，必须使用一个系统配置文件，并且该文件应命名为 sys.config 。

.app 文件中的值和系统配置文件中的值可以直接从命令行覆盖：

% erl -ApplName Par1 Val1 ... ParN ValN

示例:

% erl -ch_app file '"testlog"'
Erlang (BEAM) emulator version 5.2.3.6 [hipe] [threads:0]

Eshell V5.2.3.6  (abort with ^G)
1> application:start(test).
ok
2> application:get_env(test, file).
{ok,"testlog"}

应用程序启动类型

在启动应用程序时定义一种启动类型：

application:start(Application, Type)

application:start(Application) 与 application:start(Application, temporary) 的调用相同。类型也可以是 permanent 或 transient ：

  • 使用以下命令启动应用程序：application:start(AppName, temporary)

    正常结束：不会发生特殊情况，应用程序已停止。

    异常结束：报告错误，应用程序终止，不会重启。

  • 使用以下命令启动应用程序：application:start(AppName, transient)

    正常结束：不会发生特殊情况，应用程序已停止。

    异常结束：报告错误，所有其他应用程序都停止，VM 关闭。

  • 使用以下命令启动应用程序：application:start(AppName, permanent)

    正常结束：所有其他应用程序都终止，VM 关闭。

    异常结束：相同；所有应用程序都终止，VM 关闭。

一个应用程序可以始终通过调用 application:stop/1 显式停止。无论处于何种模式，其他应用程序都不会受到影响。

暂态模式的实际用途很小，因为当监督树终止时，原因会被设置为 shutdown ，而不是 normal 。

包含的应用程序

注意

建议不要使用包含的应用程序，原因很简单：它们严重限制了代码重用。这样想吧。我们花了大量时间来完善 ppool 的架构，使其任何人都可以使用，获得自己的池，并可以自由地使用它。如果我们将其推入一个包含的应用程序中，那么它就不能再包含在这个 VM 上的任何其他应用程序中，如果 erlcount 崩溃，那么 ppool 将与其一起被关闭，从而破坏任何想要使用 ppool 的第三方应用程序的工作。

一个应用程序可以包含其他应用程序。一个包含的应用程序有自己的应用程序目录和.app文件，但它作为另一个应用程序的监督树的一部分启动。

一个应用程序只能被一个其他应用程序包含。

被包含的应用程序可以包含其他应用程序。

没有被任何其他应用程序包含的应用程序称为一个 主要应用.

    prim_app --- app2
    prim_app --- app3
    prim_app --- app4
    prim_app --- app5

    app3 --- app31
    app3 --- app32

应用程序控制器在加载主应用程序时会自动加载任何包含的应用程序，但不会启动它们。相反，包含应用程序中的监督者必须启动包含应用程序的顶级监督者。

这意味着运行时，一个包含的应用实际上是主应用的一部分，而包含应用中的进程认为自己属于主应用。

指定包含的应用程序

包含的应用程序由included_applications键在.app文件中定义：

{application, prim_app,
 [{description, "Tree application"},
  {vsn, "1"},
  {modules, [prim_app_cb, prim_app_sup, prim_app_server]},
  {registered, [prim_app_server]},
  {included_applications, [incl_app]},
  {applications, [kernel, stdlib, sasl]},
  {mod, {prim_app_cb,[]}},
  {env, [{file, "/usr/local/log"}]}
 ]}.

启动时同步进程

包含的应用程序的监督树作为包含应用程序的监督树的一部分启动。如果需要在包含应用程序和包含应用程序之间进行进程同步，可以通过使用启动阶段来实现。

启动阶段通过start_phases键在.app文件中定义为元组列表{Phase,PhaseArgs}，其中Phase是一个原子，PhaseArgs是一个术语。

包含应用程序的 mod 键的值必须设置为 {application_starter,[Module,StartArgs]} ，其中 Module 通常是指应用程序回调模块。StartArgs 是作为 Module:start/2 回调函数参数提供的一个术语：

{application, prim_app,
 [{description, "Tree application"},
  {vsn, "1"},
  {modules, [prim_app_cb, prim_app_sup, prim_app_server]},
  {registered, [prim_app_server]},
  {included_applications, [incl_app]},
  {start_phases, [{init,[]}, {go,[]}]},
  {applications, [kernel, stdlib, sasl]},
  {mod, {application_starter,[prim_app_cb,[]]}},
  {env, [{file, "/usr/local/log"}]}
 ]}.

{application, incl_app,
 [{description, "Included application"},
  {vsn, "1"},
  {modules, [incl_app_cb, incl_app_sup, incl_app_server]},
  {registered, []},
  {start_phases, [{go,[]}]},
  {applications, [kernel, stdlib, sasl]},
  {mod, {incl_app_cb,[]}}
 ]}.

在启动具有包含应用程序的主应用程序时，主应用程序将像通常那样启动，即：

  • 应用控制器为应用程序创建一个应用程序主控
  • 应用程序主控调用 Module:start(normal, StartArgs) 来启动顶级监督者。

然后，对于主应用程序和顶级向下中的每个包含应用程序， 从左到右的顺序，应用主调用 Module:start_phase(Phase, Type, PhaseArgs) 对主应用定义的每个阶段，按照这个顺序依次调用。如果未为包含的应用定义某个阶段，则不会为该阶段和应用调用此函数。

以下要求适用于包含应用的.app文件：

  • 必须包含 {mod, {Module,StartArgs}} 选项。此选项用于查找应用程序的回调模块 Module。StartArgs 被忽略，因为仅对主应用程序调用 Module:start/2。
  • 如果包含的应用程序本身也包含其他包含的应用程序，则 必须包含 {mod, {application_starter, [Module,StartArgs]}} 选项。
  • {start_phases, [{Phase,PhaseArgs}]} 选项必须包含，并且指定的阶段集合必须是为主应用程序指定的阶段集合的子集。

当以上定义启动prim_app时，在application:start(prim_app)返回值之前，应用程序控制器会调用以下回调函数：

application:start(prim_app)
 => prim_app_cb:start(normal, [])
 => prim_app_cb:start_phase(init, normal, [])
 => prim_app_cb:start_phase(go, normal, [])
 => incl_app_cb:start_phase(go, normal, [])
ok
