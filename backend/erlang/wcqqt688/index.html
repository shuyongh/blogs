分布式Erlang

约 2869 字大约 10 分钟

2025-03-22

在分布式Erlang里，我们编写的程序会在Erlang的节点（node）上运行。节点是一个独立的Erlang系统，包含一个自带地址空间和进程组的完整虚拟机。

可以在任何节点上分裂进程，前几章讨论的所有消息传递和错误处理基本函数也都能像在单节点上那样工作。

分布式Erlang应用程序运行在一个可信环境中。因为任何节点都可以在其他Erlang节点上执行任意操作，所以这涉及高度的信任。虽然分布式Erlang应用程序可以运行在开放式网络上，但它们通常是运行在属于同一个局域网的集群上，并受防火墙保护。

rpc模块

Erlang 的 rpc 模块提供了远程过程调用（Remote Procedure Call, RPC）的功能，允许一个节点上的进程调用另一个节点上的函数

rpc 模块的主要作用是：

  1. 提供一种简单的方式，在远程节点上执行函数。
  2. 支持同步和异步调用。
  3. 提供超时机制，防止调用阻塞过久。
  4. 支持多节点并行调用。

1. rpc:call/4

本质上是在Node上执行apply(Mod, Function, Args)，然后返回结果Result，如果调用失败则返回{badrpc, Reason}。

  • 功能：同步调用远程节点上的一个函数 ，并等待结果。
  • 参数：
    • Node：远程节点名称（原子）。
    • Module：模块名称（原子）。
    • Function：函数名称（原子）。
    • Args：函数参数列表。
  • 返回值：
    • 如果调用成功，返回函数的结果。
    • 如果调用失败，返回 {badrpc, Reason}。

示例

% 调用远程节点 'node@host' 上的方法 lists:sort/1
Result = rpc:call('node@host', lists, sort, [[3, 1, 2]]).
% Result = [1, 2, 3]

2. rpc:block_call/4

  • 功能：与 rpc:call/4 类似，但在调用期间会阻塞调用者，直到远程节点完成操作。
  • 参数：同 rpc:call/4。
  • 返回值：同 rpc:call/4。

示例

Result = rpc:block_call('node@host', lists, sort, [[3, 1, 2]]).
% Result = [1, 2, 3]

3. rpc:cast/4

  • 功能：异步调用远程节点上的一个函数 ，不等待结果。
  • 参数：
    • Node：远程节点名称（原子）。
    • Module：模块名称（原子）。
    • Function：函数名称（原子）。
    • Args：函数参数列表。
  • 返回值：true（表示调用已发送）。

示例

% 调用远程节点 'node@host' 上的方法 io:format/2
rpc:cast('node@host', io, format, ["Hello, World!~n"]).

4. rpc:multicall/4

  • 功能：并行调用多个节点上的同一个函数，并收集所有节点的结果。
  • 参数：
    • Nodes：节点列表（原子列表）。
    • Module：模块名称（原子）。
    • Function：函数名称（原子）。
    • Args：函数参数列表。
  • 返回值：
    • {Results, BadNodes}：
      • Results：每个节点的调用结果列表。
      • BadNodes：调用失败的节点列表。

示例

% 在多个节点上调用 lists:sort/1
Nodes = ['node1@host', 'node2@host', 'node3@host'],
{Results, BadNodes} = rpc:multicall(Nodes, lists, sort, [[3, 1, 2]]).
% Results = [[1, 2, 3], [1, 2, 3], [1, 2, 3]]
% BadNodes = [] （如果没有失败的节点）

5. rpc:async_call/4

  • 功能：异步调用远程节点上的一个函数，并返回一个用于获取结果的键。
  • 参数：
    • Node：远程节点名称（原子）。
    • Module：模块名称（原子）。
    • Function：函数名称（原子）。
    • Args：函数参数列表。
  • 返回值：Key（用于获取结果的键）。

示例

% 在远程节点 'node@host' 上异步调用 lists:sort/1
Key = rpc:async_call('node@host', lists, sort, [[3, 1, 2]]).
% 稍后使用 rpc:yield/1 获取结果
Result = rpc:yield(Key).
% Result = [1, 2, 3]

6. rpc:yield/1

  • 功能：获取异步调用的结果。
  • 参数：
    • Key：异步调用返回的键。
  • 返回值：
    • 如果调用完成，返回函数的结果。
    • 如果调用未完成，阻塞等待结果。

示例

Key = rpc:async_call('node@host', lists, sort, [[3, 1, 2]]).
Result = rpc:yield(Key).
% Result = [1, 2, 3]

7. rpc:pmap/3

  • 功能：在多个节点上并行执行函数，并收集结果（类似于 lists:map/2 的分布式版本）。
  • 参数：
    • Nodes：节点列表（原子列表）。
    • Fun：函数（匿名函数或函数引用）。
    • ArgsList：参数列表的列表。
  • 返回值：结果列表。

示例

% 在多个节点上并行计算 lists:sort/1
Nodes = ['node1@host', 'node2@host', 'node3@host'],
Results = rpc:pmap(Nodes, fun lists:sort/1, [[3, 1, 2], [5, 4], [9, 7, 8]]).
% Results = [[1, 2, 3], [4, 5], [7, 8, 9]]

8. rpc:eval_everywhere/3

  • 功能：在多个节点上执行相同的函数。
  • 参数：
    • Nodes：节点列表（原子列表）。
    • Module：模块名称（原子）。
    • Function：函数名称（原子）。
  • 返回值：ok。

示例

% 在多个节点上执行 io:format/2
Nodes = ['node1@host', 'node2@host', 'node3@host'],
rpc:eval_everywhere(Nodes, io, format, ["Hello, World!~n"]).

global模块

global 模块是 Erlang 中用于分布式系统中全局注册和查找进程的一种机制，它允许不同节点上的进程能够通过一个全局的名字来进行通信。

global:register_name/2

  • 作用：用于将进程注册为一个全局名称。其他节点可以通过这个全局名称来查找这个进程。

  • 参数：

    • Name: 你要注册的全局名称。
    • Pid: 你要注册的进程 ID。

  • 返回值：

    • 如果注册成功，返回 true。
    • 如果该名称已经被其他进程注册，返回 false。

示例：

global:register_name(my_service, self()).

上述代码将当前进程（self()）注册为全局名称 my_service。

global:unregister_name/1

用于注销一个已经注册的全局名称。这个方法可以解除进程与全局名称的绑定。

  • 参数
    • Name: 需要注销的全局名称。

返回值：

  • 如果注销成功，返回 true。
  • 如果该名称未被注册，返回 false。

示例：

global:unregister_name(my_service).

这将注销全局名称 my_service。

global:whereis_name/1

用于根据全局名称查找对应的进程 ID。这个方法允许你查找并与跨节点的进程进行通信。

  • 参数
    • Name: 需要查找的全局名称。

返回值：

  • 如果找到进程，返回该进程的 Pid。
  • 如果没有找到该全局名称，返回 undefined。

示例：

Pid = global:whereis_name(my_service).

这将查找全局名称 my_service 对应的进程 ID。

global:send/2

用于向通过全局名称注册的进程发送消息。这是基于全局名称来发送消息，而不需要直接使用进程 ID。

  • 参数

    • Name: 目标进程的全局名称。

    • Message: 你想发送的消息。

返回值：

  • 如果发送成功，返回 true。
  • 如果全局名称未注册，返回 false。

示例：

global:send(my_service, {hello, self()}).

这将向全局名称 my_service 对应的进程发送消息 {hello, self()}。

global:transfer/2

  • **作用：**用于将一个全局名称从一个节点迁移到另一个节点。这个方法可以用于当一个节点离线时，将该节点上的全局名称迁移到另一个节点，以保证全局名称的可用性。

  • 参数

    • Name: 需要迁移的全局名称。

    • Node: 目标节点。

返回值：

  • 如果迁移成功，返回 true。
  • 如果迁移失败，返回 false。

示例：

global:transfer(my_service, 'node2@host').

将全局名称 my_service 从当前节点迁移到 node2@host。

常用方法

  • -spec spawn(Node,Mod,Func,ArgList)->Pid

    • 它的工作方式和spawn(Mod, Func, ArgList)完全一致，只是新进程是在Node上分裂的。spawn(Mod, Func, Args)会创建一个执行apply(Mod, Func, Args)的新进程。它会返回这个新进程的PID。

  • -spec disconnect_node(Node)->bool()|ignored

    • 它会强制断开与某个节点的连接

  • -spec node()->Node

    • 它会返回本地节点的名称，如果节点不是分布式的则会返回nonode@nohost。

  • -spe node(Arg)->Node

    • 它会返回Arg所在的节点。Arg可以是PID、引用或者端口。如果本地节点不是分布式的，

      则会返回nonode@nohost。

  • -spec nodes()->[Node]

    • 它会返回一个列表，内含网络里其他所有与我们相连的节点。

  • -spec is_alive()->bool()

    • 如果本地节点是活动的，并且可以成为分布式系统的一部分，就返回true，否则返回false。

消息发送

向注册了某个全局名称（RegName）的进程发送消息

{RegName, Node}!Msg

  • RegName：这是一个已经通过 global 模块注册的全局名称。通过该名称，可以在整个分布式 Erlang 系统中唯一地识别一个进程。

  • Node：这是目标进程所在的 Erlang 节点的名称，通常形式为 'node_name@host'。指定这个节点后，消息将发送到这个特定节点上运行的注册进程。

  • !：这是 Erlang 中的消息发送操作符，用于将消息发送到进程。它的右边是消息内容，左边是目标进程的进程 ID（Pid）或者全局名称。

  • Msg：这是你要发送的消息内容。它可以是任何 Erlang 语言中的数据结构，如元组、列表、整数、字符串等。

cookie保护系统

cookie系统让访问单个或一组节点变得更安全。每个节点都有一个cookie，如果它想与其他任何节点通信，它的cookie就必须和对方节点的cookie相同。

设置cookie的三种方式

cookie保护系统被设计用来创建运行在局域网（LAN）上的分布式系统，LAN本身应该受防火墙保护，与互联网隔开。跨互联网运行的分布式Erlang应用程序应该先在主机之间建立安全连接，然后再使用cookie保护系统。

方式一

在文件$HOME/.erlang.cookie里存放相同的cookie。这个文件包含一个随机字符串，是Erlang第一次在你的机器上运行时自动创建的。

这个文件可以被复制到所有想要参与分布式Erlang会话的机器上。也可以显式设置它的值

方式二

当Erlang Shell启动时，可以用命令行参数-setcookie C来把cookie设成C。

方式三

内置函数erlang:set_cookie(node(), C)能把本地节点的cookie设成原子C。

使用示例

-module(kvs).
-author("舒永好").
-compile([export_all]).
%%=====================EXPORT FUNCTION==================================

%%----------------------------------
%% 服务器：启动服务器kvs
%%----------------------------------
start()-> register(kvs,spawn(?MODULE,loop,[])).

%%----------------------------------
%% 本机：向kvs存储kv数据 
%%----------------------------------
store(Key,Value)->rpc({store,Key,Value}).

%%----------------------------------
%% 本机：查询kvs服务器上的数据 
%%----------------------------------
lookup(Key)->rpc({lookup,Key}).

%%=====================LOCAL FUNCTION==================================
%%----------------------------------
%% 本机：处理本机的消息发送和接收消息
%%----------------------------------
rpc(Req)->
  kvs!{self(),Req},
  receive
    {kvs,Reply}->Reply
  end.

%%----------------------------------
%% 服务器：处理KVS服务器的消息发送和接收消息
%%----------------------------------
loop()->
  receive
    {From,{store,Key,Value}}->
      put(Key,Value),
      From!{kvs,true},
      loop();
    {From,{lookup,Key}}->
      From!{kvs,get(Key)},
      loop()
  end.

同一个机器的不同节点上创建C/S

在同一台机器上运行两个节点时，因为它们都能访问同一个cookie文 件$HOME/.erlang.cookie，所以我们不需要在Erlang命令行里添加cookie。

  • 启动一个节点（客户端）
erl -sname client
  • 启动一个节点（服务器）
erl -sname server

同一局域网的不同机器上创建C/S

当两台机器位于同一个子网时我们也可以使用-sname。换言之，只有在不需要DNS服务的情况下，才能使用-sname。

  • 确保两个节点拥有相同的cookie。这正是启动两个节点时都使用命令行参数-setcookie abc的原因
  • 确保相关节点的完全限定主机名（fully qualified hostname）可以被DNS解析。
  • 确保两个系统拥有相同版本的代码和相同版本的Erlang
  • 启动一个节点（客户端）
erl -name client -setcookie abc
  • 启动一个节点（服务器）
erl -name server -setcookie abc
